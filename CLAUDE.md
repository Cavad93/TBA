# Правила работы над проектом «Визиторкрут»

Эти правила обязательны для соблюдения при работе над репозиторием.

## Как работать

1. **Язык общения — всегда русский.** Все ответы, пояснения, сообщения коммитов и
   комментарии для пользователя пишутся на русском языке.

2. **Выгружать в GitHub после каждого этапа.** По завершении каждого логического
   этапа (фазы, задачи) — коммит и `git push` в **рабочую ветку
   `claude/android-migration-plan-review-gbiba7`** (НЕ в `agent/android-app-migration` —
   она устарела и НЕ используется), чтобы прогресс не терялся и запускалась сборка
   APK через GitHub Actions. Подробности — в разделе «Ветки и деплой».

3. **Спрашивать, если что-то не до конца понятно.** Не додумывать и не угадывать
   в неоднозначных ситуациях — уточнять у пользователя до начала реализации.

4. **Предлагать идеи.** Если есть улучшения, альтернативы или полезные замечания —
   активно их предлагать, а не только выполнять буквально.

5. **Проверять чувствительные места реализации в сети.** Ответственные и рискованные
   решения (безопасность, платежи, персональные данные / 152-ФЗ, юридические
   моменты, редкие API и их актуальные сигнатуры) сверять с актуальными источниками
   в интернете перед внедрением.

6. **Есть доступ к боевому серверу — деплоить самому.** Разрешён SSH на VPS
   (`ssh root@45.8.228.67`, ключ уже настроен). После изменений бэкенда/сайта
   выкатывать самостоятельно: `git pull` на сервере, перезапуск сервиса
   `homevisit-api`, проверка `curl -s http://127.0.0.1:8088/api/health`. Секреты
   (пароли, ключи, токены) в чат/логи не выводить.

7. **Модульная структура, не монолиты.** Не раздувать файлы до огромных размеров
   (ориентир: ~500 строк — повод разбить; жёсткий предел — не создавать файлы
   в тысячи строк). Любой файл, независимо от типа (Kotlin/Compose, Python,
   CSS, конфиг), держать сфокусированным: один экран/модуль/ответственность —
   один файл. Новую функциональность класть в отдельные файлы/пакеты, а не
   дописывать в уже большой файл. Это упрощает будущие правки и код-ревью.

8. **Вызовы инструментов — строго в правильном формате, без обрывов.** Каждый
   вызов инструмента (Read/Edit/Bash и т.д.) оформлять ровно в требуемом синтаксисе
   и целиком. Не выдавать «сырые»/битые теги вызова, не вставлять посторонние токены
   перед вызовом, не обрывать вызов на половине. Если вызов не разобрался
   («malformed tool call») — не повторять то же самое, а переоформить корректно.
   Битый вызов подвешивает ход и прерывает работу.

## Граф кода: спрашивать граф, а не перечитывать файлы

В проекте развёрнут `codebase-memory-mcp` — индекс всего кода в виде графа: функции,
классы, файлы и рёбра между ними (кто кого вызывает, что откуда импортируется).
Структурный вопрос («кто зовёт эту функцию?», «на что повлияет правка?», «что тут
вообще за архитектура?») он закрывает за миллисекунды и ~500 токенов — там, где `grep`
и чтение файлов подряд сжигают десятки тысяч и всё равно могут не найти.

Подробности, ловушки и таблица «вопрос → инструмент» — в скилле
[.claude/skills/code-graph/SKILL.md](.claude/skills/code-graph/SKILL.md). Он подхватывается
автоматически; читать его целиком, когда дело касается графа. Здесь — только то, что
обязано быть на виду:

**Проектов в графе два, и это не случайность.** Бэкенд индексируется от
`home_visit_profit_bot/`, Android-клиент — от `home_visit_profit_bot/android_location_client/`.
Корень репозитория `TBA/` одним проектом **не индексировать**:

- Корень индексации обязан совпадать с корнем импорта. Импорты бэкенда выглядят как
  `from app.services… import …`, а `app/` лежит внутри `home_visit_profit_bot/`.
  Проиндексируешь от `TBA/` — межмодульные вызовы Python **молча теряются**: `find_zone`
  и `get_distance_matrix` показывали ноль вызывающих при живых вызовах в коде.
- Один язык — один граф. В общем графе индексатор связывает вызовы **по совпадению
  имени** и придумывает рёбра между языками: он утверждал, что Compose-компонент
  `AppSettingsCard` вызывает питоновский `profile_service.snapshot()`. Отсекается файлом
  `home_visit_profit_bot/.cbmignore`.

**Граф — карта, а не территория.** «Нет вызывающих» не значит «мёртвый код»: всё, что
дёргает фреймворк (pytest, таблица маршрутов в `location_api.py`, рантайм Compose),
вызывающих в графе не имеет и выглядит мёртвым. Граф хорош, чтобы быстро понять, **куда
смотреть**; решение об удалении или рефакторинге принимается по коду, после проверки
грепом. Инструмент, который уверенно отвечает неправильно, опаснее того, который честно
отвечает «не знаю» — это ровно тот урок, что и с OSRM, вернувшим 0 км вне покрытия.

## Ветки и деплой

- Рабочая ветка проекта — `claude/android-migration-plan-review-gbiba7` (не
  `agent/android-app-migration`, она устарела). Пуш в рабочую ветку запускает
  сборку APK в GitHub Actions.
- Код на сервере: `/opt/tba` (репозиторий под пользователем `homevisit`), сервис
  `homevisit-api` (systemd), API слушает `127.0.0.1:8088` за Caddy. Деплой бэкенда:
  ```
  ssh root@45.8.228.67 'cd /opt/tba && \
    sudo -u homevisit git fetch origin && \
    sudo -u homevisit git checkout claude/android-migration-plan-review-gbiba7 && \
    sudo -u homevisit git pull && \
    systemctl restart homevisit-api && sleep 2 && \
    curl -s http://127.0.0.1:8088/api/health'
  ```
- Сайт: `/opt/tba/home_visit_profit_bot/deploy/site` (Caddy отдаёт статику).

## Архитектура: два сервера, и это не симметрия

**Сервер 1 — приложение.** `45.8.228.67`, RUVDS. PostgreSQL (RLS), Python-бэкенд
(`homevisit-api`), Caddy, сайт. Больше на нём ничего быть не должно: ни OSRM,
ни выгрузок OSM, ни тяжёлых сборок. Диск 20 ГБ, и он не резиновый.

**Сервер 2 — карты и данные.** `194.87.93.174`, Ubuntu 24.04, 4 ядра / 16 ГБ RAM
(+ 32 ГБ свопа) / 100 ГБ SSD. Здесь живут: OSRM (Docker), выгрузка
`/opt/osrm/data/russia5.osm.pbf` (5 федеральных округов, склеены `osmium merge`),
граф маршрутов и фабрика зон платной парковки.

**Кто кого спрашивает — важно не перепутать:**

- **Маршруты**: сервер 1 → HTTP → OSRM на сервере 2. Настройка `osrm_url`.
  Если сервер 2 лёг, у нас есть запасной расчёт по прямой (`routing_fallback_to_estimate`).

- **Зоны парковки: НАОБОРОТ.** Сервер 2 их только ПРОИЗВОДИТ (osmium разбирает ту же
  выгрузку, что съел OSRM), а хранит и ищет их сервер 1 — у себя, локально.
  Причина: проверка «в платной ли зоне» дёргается на КАЖДОЙ точке GPS (раз в 30–60 с
  на каждого работника в смене) и на каждой оценке заказа. Сейчас это ~1 мс по индексу
  в PostgreSQL. Сделать её сетевым запросом значит: (а) добавить round-trip в самый
  горячий путь продукта, (б) уронить приём GPS у ВСЕХ, когда сервер 2 недоступен.
  Маршруты такое переживут, приём GPS — нет.

**Три профиля OSRM, у каждого свой порт.** Это не прихоть: OSRM игнорирует профиль
в адресе запроса и отдаёт то, какой граф загрузил. Спросить пеший маршрут
у автомобильного инстанса — значит получить автомобильный ответ и не заметить подмены.

| Профиль | Покрытие | Порт | Переменная |
|---|---|---|---|
| `car` | 5 округов | 5000 | `OSRM_URL` |
| `foot` | Москва и Петербург | 5002 | `OSRM_URL_FOOT` |
| `bicycle` | Москва и Петербург | 5003 | `OSRM_URL_CYCLING` |

Легковая, кроссовер, Газель, грузовик, мотоцикл — всё это `driving`.

**ГЛАВНОЕ ПРО ГРАНИЦЫ ПОКРЫТИЯ, не наступить второй раз.** OSRM «прилипает» к ближайшей
точке своего графа БЕЗ ограничения по расстоянию. Спрошенный про Нижний Новгород, пеший
маршрутизатор притянул обе точки к одному узлу на краю Москвы — за 400 км — и вернул код
`Ok`, **0 км и 0 минут**. Приложение приняло бы это за правду: заказ без дороги выглядит
бесконечно выгодным, вердикт «стоит ехать» на поездку в четыреста километров.

Поэтому в каждом запросе идёт `radiuses` (см. `routing_service.SNAP_RADIUS_M`), а
`NoSegment` от OSRM превращается в `OutsideCoverageError` — и человеку говорят словами:
«этот адрес пока вне покрытия наших карт, введите километры вручную».

Ловушка внутри ловушки: `NoSegment` приходит с кодом **HTTP 400**. Звать
`raise_for_status()` нельзя — тела не прочитаем и не отличим «человек вне покрытия»
от «сервер недоступен». Первое надо сказать человеку, второе — молча посчитать по прямой.

**Обновление карты** — `/opt/osrm/refresh.sh`, его раз в два месяца зовёт GitHub Actions
(`.github/workflows/parking-refresh.yml`, файл живёт в `main`: расписание с других веток
GitHub не запускает). Скрипт качает свежую выгрузку, пересобирает все три графа и зоны
парковки из неё же, строит в отдельной папке и подменяет готовое одним движением —
упавшая сборка не роняет работающие маршрутизаторы.

Сборка графа: `osrm-extract` берёт пик ~14,4 ГБ — влезает в 16 ГБ, своп на 32 ГБ нужен
как страховка. Дальше `osrm-partition` + `osrm-customize` (схема MLD).

## Публикация APK на сайт (обязательно после каждой сборки)

**После того как GitHub Actions успешно собрал APK — его нужно выложить на сайт
для скачивания.** CI сам на сайт НЕ выкладывает: сайт отдаёт файл
`https://vizitorkrut.ru/vizitorkrut.apk` из папки статики, и его надо обновлять
вручную, иначе пользователь скачает старую версию. Порядок:

1. Дождаться зелёной сборки `Build Android APK` и скачать артефакт релиза:
   ```
   gh run download <RUN_ID> -n vizitorkrut-release-apk -D <tmp>
   ```
2. Скопировать на сервер и заменить файл сайта (владелец — `homevisit`),
   сохранив бэкап старого:
   ```
   scp <tmp>/app-release.apk root@45.8.228.67:/tmp/vizitorkrut-new.apk
   ssh root@45.8.228.67 'D=/opt/tba/home_visit_profit_bot/deploy/site; \
     cp -a "$D/vizitorkrut.apk" "$D/vizitorkrut.apk.bak-$(date +%Y%m%d-%H%M%S)"; \
     install -o homevisit -g homevisit -m 0644 /tmp/vizitorkrut-new.apk "$D/vizitorkrut.apk" && \
     rm -f /tmp/vizitorkrut-new.apk && sha256sum "$D/vizitorkrut.apk"'
   ```
3. Проверить, что сайт отдаёт свежий файл (сверить `content-length`/`sha256`
   локального артефакта и ответа `curl -sI https://vizitorkrut.ru/vizitorkrut.apk`).

`versionCode` берётся из номера CI-сборки (монотонно растёт) — обновление
устанавливается «поверх», ручной бамп не нужен.

## Контекст проекта

- Продукт: «Визиторкрут» (`vizitorkrut.ru`) — массовое многопользовательское
  приложение для выездных работников (врачи, таксисты, курьеры, мастера на дом).
- Девиз: «Навигатор показывает, куда ехать. Наше приложение показывает — стоит ли ехать».
- Бэкенд + сайт: Ubuntu VPS, PostgreSQL с изоляцией данных по пользователю (RLS),
  Caddy (авто-HTTPS), корпоративная почта Yandex 360.
- Android-клиент: Kotlin/Compose, сборка APK через GitHub Actions.
