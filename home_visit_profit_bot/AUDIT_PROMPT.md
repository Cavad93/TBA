# ЗАДАЧА: автономный аудит и укрепление «Визиторкрут»

> Промпт для новой AI-сессии. Вставлять целиком, как есть.

Ты работаешь над проектом в `TBA/home_visit_profit_bot`. Твоя цель — **не строить новое,
а довести до безупречности то, что уже есть**: проверить каждую функцию, каждую ветку
логики, каждый расчёт; найти ошибки, неточности и слабые места; исправить их; доказать
тестами.

## Железные правила

1. **СТРОГО соблюдай `TBA/CLAUDE.md`** во всём: язык только русский (ответы, коммиты,
   комментарии); модульные файлы (~500 строк — повод разбить); структурные вопросы по
   коду задавай графу кода (code-graph), а не перечитывай файлы подряд; многострочные
   скрипты — файлом, не heredoc'ом.
2. **НОВЫХ ФИЧ НЕ ДОБАВЛЯТЬ.** Только доработка, исправление и улучшение существующего.
   Появилась идея новой фичи — она идёт в файл предложений, а не в код.
3. **Чувствительные места сверяй с актуальными источниками в интернете ПЕРЕД правкой**:
   безопасность, платежи, персональные данные / 152-ФЗ, юридические моменты, редкие API
   и их текущие сигнатуры. Библиотека/API могли измениться — проверяй, а не помни.
4. **Ошибку не обходи и не замалчивай**: воспроизведи, найди причину, при необходимости
   изучи в сети лучший способ починки, почини, закрой тестом, подтверди зелёным прогоном.
5. **Не ломай работающее.** Любая правка сопровождается тестом, доказывающим, что старое
   поведение не пострадало.
6. **Держи граф кода свежим.** Ты правишь код — значит индекс `codebase-memory` устаревает,
   а устаревший граф уверенно отвечает неправильно (это опаснее, чем «не знаю»). Правила:
   - **В начале этапа** — `detect_changes`, и если есть расхождения → `index_repository`.
   - **После каждой порции правок** (перед коммитом) — переиндексируй затронутый проект.
   - **Проектов ровно два, не путать:** бэкенд индексируется от `home_visit_profit_bot/`,
     Android-клиент — от `home_visit_profit_bot/android_location_client/`.
   - **Корень `TBA/` одним проектом НЕ индексировать никогда** — корень индексации обязан
     совпадать с корнем импорта, иначе межмодульные вызовы Python молча теряются, а
     индексатор придумывает связи между Kotlin и Python по совпадению имён (см. CLAUDE.md).

## Два файла — твоя память между этапами

### 1. `home_visit_profit_bot/AUDIT_JOURNAL.md` — журнал аудита

Создай, если нет. **Читай его ПЕРВЫМ делом перед каждым новым этапом** — он говорит,
что уже проверено, что нашёл и что делать дальше.

Структура каждой записи:

```
## Этап N — <что проверял> — <дата>
**Гипотеза:** что подозревал и почему.
**Метод:** как проверял (тесты/чтение/эксперимент/источники в сети).
**Результат:** подтвердилось / опровергнуто / найдено.
**Правки:** что изменил (файлы, коммит).
**Доказательство:** какие тесты зелёные, что проверено на проде.
**Следующая задача:** <конкретно, что делать на этапе N+1 и почему именно это>
```

Отдельный раздел вверху файла: **`## ОТКРЫТЫЕ ОШИБКИ (разбирать в первую очередь)`** —
список найденных, но ещё не исправленных дефектов с приоритетом (критично/важно/мелочь)
и коротким описанием. Нашёл ошибку по ходу другой задачи — **немедленно запиши сюда**,
даже если сейчас чинишь другое. При постановке новой задачи **сначала берёшь отсюда**,
пока список не пуст.

### 2. `home_visit_profit_bot/AUDIT_PROPOSALS.md` — предложения на одобрение

Сюда пишешь то, что **не делаешь сам**, а предлагаешь Джаваду:

- улучшения, которые стоило бы внедрить, но они меняют поведение/архитектуру/UX;
- идеи новых фич, возникшие по ходу аудита;
- всё, что требует его решения, денег, ключей или внешних действий.

Формат: `### Предложение — <название>` → **Проблема / Что предлагаю / Зачем (выгода) /
Цена вопроса (риск, объём) / Статус: ждёт одобрения**. В код это НЕ идёт, пока Джавад
не одобрит.

## Цикл работы (выполняй бесконечно, автономно)

1. **Прочитай `AUDIT_JOURNAL.md`** целиком: что уже проверено, какие ошибки открыты,
   какая задача поставлена на этот этап.
2. **Проверь свежесть графа** (`detect_changes`) и переиндексируй, если код разошёлся с
   индексом. Спрашивать граф по устаревшему индексу — значит получать уверенную ложь.
3. **Выбери задачу:** если в «ОТКРЫТЫХ ОШИБКАХ» что-то есть — берёшь оттуда самое
   приоритетное. Пусто — берёшь задачу, поставленную прошлым этапом.
4. **Выдвини гипотезу** и проверь её: тестами, чтением кода через граф, экспериментом
   на реальных данных, сверкой с источниками в сети. Опровергнутая гипотеза — тоже
   результат, её тоже записываешь.
5. **Исправь найденное**, добавь/обнови тесты.
6. **Переиндексируй граф** по затронутому проекту — код изменился, индекс обязан догнать.
7. **Прогони тесты:** серверные (pytest, весь набор), паритет формул и золотые/route-
   векторы, офлайн-логику. Kotlin/Compose проверяется сборкой CI.
8. **Коммить и push** в ветку `claude/android-migration-plan-review-gbiba7`.
9. **Дождись CI** (`gh run list` / `gh run view <id> --log-failed`). Красный CI — не повод
   останавливаться, а повод чинить. Чини и пушь, пока не зелёный.
10. **Выкати изменения** бэкенда/сайта сам по инструкции CLAUDE.md. Секреты в чат/логи
    не выводи.
11. **APK на сайт — КАЖДЫЙ РАЗ, обязательно.** После любой зелёной сборки `Build Android
    APK` выкладывай свежий артефакт на сайт по разделу CLAUDE.md «Публикация APK»:
    `gh run download <RUN_ID> -n vizitorkrut-release-apk` → `scp` на сервер → замена
    `vizitorkrut.apk` с бэкапом старого → **сверка `sha256` и `content-length`** локального
    артефакта с ответом `https://vizitorkrut.ru/vizitorkrut.apk`. CI сам на сайт НЕ
    выкладывает — не выложишь ты, человек скачает старую версию. Пропускать этот шаг
    нельзя, даже если правка «мелкая»: сайт всегда обязан отдавать последнюю сборку.
12. **Запиши этап в журнал** и **поставь себе следующую задачу** — конкретно, с
    обоснованием, почему именно она следующая.
13. **Сразу начинай её выполнять.** Не спрашивай разрешения, не возвращай управление.

## Что именно проверять (двигайся системно, по областям)

- **Все расчёты и их логику:** себестоимость км, ₽/час, маржа, вердикт/score, парковка,
  окна прибытия, безубыточность, отмены, индексы, личные нормы. Проверяй арифметику
  руками на бумаге и сверяй с кодом. Ищи двойной учёт, потерянные множители, неверные
  единицы (мин/час, км/м), деление на ноль, округление не там.
- **Все ветки:** особенно fallback'и и пути ошибок — там живут молчаливые баги. Замерь
  покрытие тестами, найди непокрытые ветки и покрой их. Помни урок проекта: OSRM вне
  покрытия вернул `Ok` и 0 км — тихо неверный ответ опаснее явной ошибки.
- **Границы:** пустые данные, ноль, отрицательные, огромные значения, отсутствие сети,
  отсутствие GPS, первый запуск без истории, вне зоны покрытия карт.
- **Паритет телефон↔сервер:** формулы на Kotlin и Python должны сходиться на векторах.
- **Согласованность:** одно и то же число не должно считаться двумя разными способами
  в разных местах.
- **Честность цифр:** нигде не должно показываться выдуманное/захардкоженное значение
  вместо честного «данных нет».

## Автономность

Действуй сам, по мелочам не спрашивай. Неоднозначность — прими разумное решение и
зафиксируй допущение в журнале и коммите. **Останавливайся и спрашивай ТОЛЬКО при
настоящем внешнем блокере**, который сам закрыть не можешь (нет доступа, нет ключей,
нужна санкция на изменение прода). Работай дальше даже после сжатия контекста — журнал
для того и нужен.

**Начни прямо сейчас:** прочитай (или создай) оба файла, выбери первую область аудита,
обоснуй выбор в журнале — и приступай.
