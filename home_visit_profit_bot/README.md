# Telegram-бот рентабельности вызовов врача на дом

MVP использует бесплатные инструменты:

- Nominatim/OpenStreetMap для геокодинга адресов.
- OSRM endpoint для расчёта маршрутов и матриц.
- Локальную TSP-оптимизацию порядка адресов без платных API.
- Встроенный no-Docker fallback по координатам, если OSRM недоступен.
- Личную поправку времени дороги: бот сравнивает факт дня с OSRM и постепенно корректирует минуты маршрута.

Если адрес не найден по улице и дому, бот попросит координаты. Ручной ввод километров и минут дороги остаётся запасным режимом на случай, если OSRM недоступен.

Бот не хранит ФИО пациентов, диагнозы, телефоны и медицинские комментарии. В базе есть только адрес/локация, финансовые суммы, технические параметры маршрута и статусы.

## Быстрый старт

```bash
cd home_visit_profit_bot
python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Откройте `.env` и укажите токен:

```text
TELEGRAM_BOT_TOKEN=ваш_токен_бота
```

Запуск:

```bash
python -m app.main
```

SQLite-база создастся автоматически в `data.sqlite3`.

## Основной сценарий

Начать день:

```text
/newday
```

Добавить кандидат:

```text
/add Мурино, Воронцовский бульвар 5 | 2500
```

После цены бот спросит, от какой клиники вызов:

```text
1) Династия
2) ПСК
3) ВИТАМЕД
4) ДНД
```

Можно ответить цифрой или названием клиники.

Полная форма, если нужно задать ручной fallback:

```text
/add адрес | доход | км | минуты дороги | район | клиника
```

Если бот просит километры или минуты вручную, он показывает контекст:

```text
Откуда: текущая точка маршрута
Куда: новый адрес
```

Текущая точка — это старт дня, если завершённых адресов ещё нет, или последний завершённый через `/complete` адрес. Если вы вручную учитываете перестройку всего оставшегося маршрута, вводите добавочное значение для дня.

Если адрес не найден, бот спросит координаты:

```text
59.9386, 30.3141
```

Если координаты введены как `30.3141, 59.9386`, бот распознает типичный для Петербурга порядок `lon, lat` и сам поменяет местами.

Если OSRM не вернул маршрут, бот попросит километры и минуты дороги для ручного расчёта.
Если координаты есть и включён `routing.fallback_to_estimate`, бот сам рассчитает оценочный маршрут без OSRM: прямая дистанция × `routing.straight_line_factor`, время по средней скорости дня.

Когда день закрывается через `/endday`, бот считает:

```text
поправка OSRM = фактическое время дороги / плановое время дороги OSRM
```

Например, если OSRM планировал 60 минут, а по факту дорога заняла 90 минут, поправка будет `x1.50`. Следующие дни будут использовать среднюю поправку за последние дни: километры останутся OSRM-километрами, но время дороги станет реалистичнее для вашей практики.

Принять или отклонить последний кандидат:

```text
/accept
/reject
```

Завершить адрес:

```text
/complete 1
```

`/done 1` тоже работает как алиас.

Учесть деньги в течение дня:

```text
/telemed 3500
/parking 450
/food 700
/fuel_compensation 500
/parking_compensation 300
/toll 250
/toll_compensation 250
/compensation 1200
/expense 300 мойка
```

`/fuel_compensation` — компенсация топлива. `/parking_compensation` — компенсация парковки. `/compensation` оставлена для прочих компенсаций от клиник.
`/toll` — расход на платную дорогу. `/toll_compensation` — компенсация платной дороги.

Телемедицина добавляется без адреса и без дороги. По умолчанию одна телемедицинская консультация занимает 3 минуты из `defaults.telemed_minutes`:

```text
/telemed 700
```

Основной способ в Telegram — кнопка `Телемедицина` в меню. Бот спросит сумму, затем покажет кнопки клиники:

```text
ПСК
ДНД
```

У других клиник телемедицина не используется.

Если нужно указать другое время:

```text
/telemed 700 5 ПСК
```

Эти минуты учитываются в доходе за час, но не попадают в среднее время обычного выезда на адрес.

Посмотреть маршрут и сводку:

```text
/route
/summary
/stats
```

`/summary` и `/stats` показывают общий грязный доход, общий чистый доход и отдельную разбивку по клиникам: грязный доход, рассчитанный чистый доход, рабочее время и чистый доход в час. Общие расходы распределяются по клиникам пропорционально рабочему времени клиники за период.

`/route` показывает только активные незавершённые адреса. Адреса, завершённые через `/complete`, уходят из активного маршрута, а оставшийся маршрут переоптимизируется от последней завершённой точки.

Статистика:

```text
/stats
/stats day 2026-07-07
/stats month 2026-07
/stats year 2026
```

`/stats` без параметров показывает текущий месяц. В отчёте есть грязный доход, чистый доход, топливо, амортизация, компенсации топлива и парковки, расходы на парковку, еду и напитки, количество адресов, километраж, чистый доход за час, доход на адрес и доход на километр.

Если геокодер нашёл не тот объект, в расчёте будет видно:

```text
Найдено как: ...
Координаты: ...
Участок до нового адреса:
- откуда: ...
- куда: ...
```

Сбросить кэш адреса:

```text
/clear_address_cache Кушелевская дорога 5к2
```

Сбросить весь кэш:

```text
/clear_address_cache all
```

Завершить день пошаговым диалогом:

```text
/endday
```

Вечерний чек-лист спрашивает вручную:

- общий километраж;
- вечернее показание одометра; утреннее показание бот берёт из `/newday`;
- количество завершённых адресов;
- сколько часов работали;
- сколько часов были в дороге;
- сколько потратили на топливо, и сколько литров заправили, если заправлялись;
- фактический расход топлива л/100 км за день, вручную; можно ввести `-` для среднего значения;
- доход и минуты телемедицины;
- еду и напитки;
- парковку и компенсацию парковки;
- компенсацию топлива;
- платные дороги и компенсацию платных дорог;
- прочие компенсации и прочие расходы.

В `/newday` бот спрашивает текущее показание одометра. Если уже есть вчерашнее вечернее показание, можно написать `-`, и бот возьмёт его как утреннее. Вечером в `/endday` бот спросит текущее показание и сам посчитает пробег за день:

```text
пробег по панели = вечерний одометр - утренний одометр
```

Часы можно вводить как `8.5` или как `8:30`. Если сегодня не заправлялись, в вопросе про топливо введите `0`: бот возьмёт последнюю цену литра. Расход л/100 км он всё равно спросит отдельным вечерним вопросом: введите фактическую оценку или `-`, чтобы взять среднее.

Формула транспорта:

```text
цена литра = сумма заправки / литры
топливо ₽/км = цена литра × расход л/100 км / 100
амортизация ₽/км = топливо ₽/км × 0.8
транспорт ₽/км = топливо ₽/км + амортизация ₽/км
```

Если заправили 60 л на 4300 ₽, а расход 10 л/100 км:

```text
цена литра = 4300 / 60 = 71.67 ₽/л
топливо = 71.67 × 10 / 100 = 7.17 ₽/км
амортизация = 7.17 × 0.8 = 5.73 ₽/км
транспорт = 12.90 ₽/км
```

Если в этот день было 20 рабочих км, на работу спишется:

```text
20 × 12.90 = 258 ₽ транспортных расходов
```

Заправка не списывается вся в день покупки. Она только обновляет цену литра. Расход л/100 км бот берёт из ручного вечернего ввода, поэтому каждый день списывается расчётный расход по рабочему пробегу.

Например, если утром одометр был `291345`, а вечером `291385`, бот сам возьмёт пробег дня `40 км`.

## GPS-уведомления с Android

В проект добавлен простой Android-клиент: `android_location_client`.

Как работает:

- утром вы запускаете рабочий день командой `/newday`;
- в Android-приложении нажимаете `Начать отправку GPS`;
- приложение отправляет координаты на сервер бота;
- сервер смотрит только активный рабочий день и адреса со статусом `accepted`;
- если вы находитесь рядом с адресом дольше заданного времени и средняя скорость за последние 12 минут ниже 10 км/ч, бот присылает уведомление в Telegram;
- адрес закрывается только после вашего нажатия `Да, закрыть`;
- если вы не ответили на уведомление, система не зависает: GPS-приложение продолжает отправлять точки, а бот продолжает фоновый учёт;
- после `/endday` активного дня нет, поэтому GPS-точки больше не приводят к уведомлениям.

GPS-учёт работает параллельно ручному учёту:

- приложение отправляет координаты примерно раз в минуту;
- если точка за минуту “прыгнула” больше чем на 10 км, такой сегмент считается плохим GPS-сигналом и не участвует в скорости;
- если средняя скорость за 12 минут выше 10 км/ч, бот считает, что вы в движении;
- если вы рядом с адресом и скорость последнего валидного сегмента ниже 10 км/ч, бот начинает считать стоянку сразу с этой минуты;
- 12 минут используются как подтверждение стоянки перед уведомлением, а не как начало отсчёта;
- если средняя скорость за 12 минут остаётся ниже 10 км/ч и вы рядом с адресом больше 12 минут, бот считает это временем на адресе и присылает подсказку;
- если вы рядом с точкой финиша больше 30 минут и скорость ниже 10 км/ч, бот считает, что рабочая поездка завершена по GPS.

В вечернем `/endday` бот предложит GPS-оценки:

- сколько всего длился рабочий день от начала движения до финиша;
- сколько времени было в пути;
- какое среднее время на адресе получилось по GPS.

На вопросы про часы можно ответить `-`, чтобы взять GPS-оценку, или ввести своё значение вручную.

GPS-оценка времени в пути используется для личной поправки OSRM: если в `/endday` взять GPS-время дороги через `-`, бот сравнит его с чистым OSRM-планом дня и сохранит коэффициент `факт / OSRM`. В следующих днях OSRM-километры останутся без изменений, а минуты маршрута будут умножаться на средний личный коэффициент.

Для GPS-дороги бот считает не только быстрые сегменты, но и медленное движение вне геозон адресов: пробки, светофоры и плотный поток попадают во фактическое время пути. Стоянки у распознанных адресов вычитаются как время на адресе.

Финиш можно поменять в течение дня:

```text
/finish
```

Или нажать кнопку `Изменить финиш` в Telegram-меню.

По умолчанию:

```yaml
location_api:
  enabled: true
  host: "0.0.0.0"
  port: 8088
  geofence_radius_m: 120
  dwell_minutes: 12
  notification_cooldown_minutes: 60
```

В `.env` задайте длинный секретный ключ:

```env
LOCATION_API_ENABLED=true
LOCATION_API_HOST=0.0.0.0
LOCATION_API_PORT=8088
LOCATION_API_KEY=replace_with_long_random_key
```

Команда `/start` сохраняет текущий Telegram-чат для GPS-уведомлений. Выполните её один раз после запуска бота.

На сервере должен быть открыт порт `LOCATION_API_PORT`, например `8088`. В Android-приложении укажите:

```text
URL сервера: https://ваш-домен:8088/location
API ключ: значение LOCATION_API_KEY
```

Если используете HTTP без HTTPS, Android может заблокировать соединение на новых версиях. Лучше поставить nginx/Caddy с HTTPS перед портом `8088`.

Сборка APK:

1. Загрузите проект в GitHub.
2. Откройте `Actions`.
3. Запустите workflow `Build Android APK`.
4. Скачайте артефакт `home-visit-gps-debug-apk`.

## Настройки

```text
/settings
/set_car_cost 17.05
/set_fuel_price 70
/set_fuel_consumption 10
/set_amortization_factor 0.8
/set_min_hourly 600
/set_home Дом
/set_default_start Дом
/set_default_finish Дом
/set_base_districts Приморский, Выборгский, Калининский
/set_osrm_url https://router.project-osrm.org
/set_routing_fallback true
/set_straight_line_factor 1.35
/set_route_time_factor 1.25
/set_min_marginal_hourly 600
/set_outside_min_hourly 900
/set_outside_extra 500
```

Основные действия также доступны кнопками в Telegram:

- `Начать день`
- `Добавить адрес`
- `Принять`
- `Отклонить`
- `Маршрут`
- `Завершить адрес`
- `Отменить адрес`
- `Изменить финиш`
- `Завершить день`
- `Сводка`
- `Статистика`
- `Настройки`

`/set_fuel_price` задаёт цену литра, `/set_fuel_consumption` — расход л/100 км. `/set_car_cost` оставлен как совместимая настройка ₽/км для первых расчётов, пока нет топливной статистики. Амортизация считается как `топливо * amortization_factor`, по умолчанию `0.8`.

Для расчёта спецтарифа:

- `/set_min_hourly` — минимальная чистая доходность всего дня;
- `/set_min_marginal_hourly` — минимальная маржинальная доходность самого нового адреса;
- `/set_outside_min_hourly` — минимальная чистая доходность дня для адресов вне базовой зоны;
- `/set_outside_extra` — минимальная фиксированная надбавка вне базовой зоны.

Для адреса вне базовой зоны бот требует, чтобы после добавления адреса чистая доходность в час не стала ниже текущей. Минимальная надбавка считается как максимум из требований: сохранить минимум дня, не снизить текущую доходность вне зоны, окупить сам адрес по маржинальной ставке и покрыть фиксированную надбавку вне зоны.

Если пациент или страховая отменили вызов, используйте кнопку `Отменить адрес`. Она отменяет текущий первый активный адрес. Для конкретного номера можно использовать:

```text
/cancel_visit 2
```

## Запуск без Docker

Docker не обязателен. Бот может работать как обычное Python-приложение на сервере:

```bash
cd home_visit_profit_bot
python3.11 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
python -m app.main
```

В `config.yaml` по умолчанию стоит публичный OSRM:

```yaml
routing:
  osrm_url: "https://router.project-osrm.org"
  fallback_to_estimate: true
  straight_line_factor: 1.35
```

Если публичный OSRM недоступен или вы не хотите на него полагаться, оставьте `fallback_to_estimate: true`. Тогда бот продолжит работать по координатам без Docker и без локального OSRM. Точность будет ниже, но маршруты будут оптимизироваться стабильно.

## Что намеренно упрощено в MVP

- Можно использовать публичный OSRM или встроенный оценочный fallback без Docker. Для максимальной точности на высокой нагрузке позже можно поднять отдельный OSRM-сервер и поменять `routing.osrm_url`.
- Административный район определяется по данным Nominatim и локальной эвристике для базовых районов.
- Завершённые адреса не содержат медицинских данных и не требуют данных пациента.

Позже в модулях `services/geocoding_service.py`, `services/routing_service.py` и `services/optimization_service.py` можно заменить публичные endpoints на локальные сервисы или OR-Tools.
