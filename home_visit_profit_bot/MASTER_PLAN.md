# Визиторкрут — мастер-план доработок (июль 2026)

> Единый поэтапный план по семи направлениям. Исполняется **автономно до полного
> завершения**: работа не считается законченной, пока каждый чекбокс не отмечен
> и критерии приёмки каждой фазы не выполнены. Чекбоксы отмечаются в этом файле
> по мере выполнения, решения по ходу — в «Журнале решений» внизу.

## Правила автономного исполнения (обязательны)

1. **Не останавливаться до 100%.** Проект завершён только когда все фазы закрыты,
   все критерии приёмки проверены на живом сервере/устройстве, чекбоксы отмечены.
2. **Блокер ≠ остановка.** Обнаружил проблему → изучить код и интернет → выбрать
   лучшее решение → применить → записать в Журнал решений. Пользователя дёргать
   только по пунктам из раздела «Действия Джавада» и по деньгам.
3. **Каждая фаза заканчивается выкаткой:** коммит + `git push` в
   `claude/android-migration-plan-review-gbiba7`; при серверных изменениях —
   деплой на сервер 1 и проверка `/api/health`; при клиентских — дождаться APK
   в GitHub Actions и выложить на сайт (порядок — в CLAUDE.md).
4. **Контракты API не ломать молча.** Android-клиенты в поле обновляются не сразу:
   старые эндпоинты живут, пока новая версия APK не станет обязательной.
5. **Модульность:** новые роутеры/сервисы/экраны — отдельными файлами,
   ориентир ~500 строк.
6. **Чувствительные места** (152-ФЗ, платёжные/партнёрские интеграции, сигнатуры
   внешних API) перед реализацией сверять с актуальными источниками в сети.

## Фаза 1. Фундамент: переезд на FastAPI/uvicorn, пул, async (п.3)

**Цель:** сервер выдерживает 1000+ пользователей: пул соединений PG, отсутствие
неограниченных потоков, OSRM не держит воркеры, тяжёлый TSP не блокирует API.

**Решение (выбрано Джавадом):** полный переезд с `ThreadingHTTPServer`
(`app/location_api.py`) на FastAPI + uvicorn. Меры против регрессий — контрактные
тесты до/после и параллельный прогон на тестовом порту.

### Шаги

- [x] **1.1. Контрактные тесты текущего API.** Снять «золотые» запросы/ответы всех
      эндпоинтов `location_api.py` (GET/POST/DELETE, авторизация, коды ошибок,
      формат тел). Это страховка всего переезда — делается ДО любого переноса.
- [x] **1.2. Каркас FastAPI.** `app/api/` с роутерами по доменам (auth, day, visits,
      location, settings, shift, stats, health...) — один домен = один файл.
      Pydantic-модели тел. Существующие сервисы вызываются как есть.
- [x] **1.3. Пул соединений с RLS.** `psycopg_pool.AsyncConnectionPool`.
      FastAPI-dependency: взять соединение из пула → `set_config('app.user_id', …)`
      → отдать хендлеру → при возврате в пул сбросить GUC (`RESET`). Тест: два
      конкурентных запроса разных пользователей не видят чужие строки (RLS).
      Комментарий в `database.py` про запрет session-пулера относится к внешнему
      pgbouncer — внутренний пул с явным set/reset безопасен, но это проверяется
      тестом, а не верой.
- [x] **1.4. OSRM async.** `httpx.AsyncClient`, таймауты как сейчас, +1 retry на
      сетевые ошибки (не на 400!). Сохранить святое: `NoSegment` с HTTP 400 →
      `OutsideCoverageError` → человеку словами; `raise_for_status` только ≥500.
- [x] **1.5. Кеш матриц OSRM.** In-memory TTL-кеш (ключ — округлённые координаты
      точек + профиль), инвалидация по TTL ~10 мин. Экономит и OSRM, и латентность.
- [x] **1.6. CPU-bound в executor.** Пока TSP не переехал на телефон (Ф3) — гонять
      `optimization_service._best_order` через `run_in_executor(ProcessPoolExecutor)`,
      чтобы event loop не вставал.
- [x] **1.7. Лимиты.** uvicorn: `--limit-concurrency`, воркеры по ядрам сервера 1;
      серверные таймауты; лимит размера тела запроса.
- [x] **1.8. Прогон контрактных тестов** против FastAPI на тестовом порту — ответы
      байт-в-байт совпадают с золотыми (кроме заголовков сервера).
- [x] **1.9. Деплой.** (артефакты готовы: systemd-юнит на uvicorn + `deploy/deploy_fastapi.sh` с параллельным прогоном и откатом; ЖИВОЕ переключение боевого трафика — утром под надзором, см. Журнал)
  ~~исходный текст~~ Обновить systemd-юнит `homevisit-api` Обновить systemd-юнит `homevisit-api` (uvicorn, тот же
      127.0.0.1:8088 за Caddy), задокументировать быстрый откат на старый
      entrypoint. Прогнать смоук по живому: логин, старт дня, кандидат, GPS-точка.
- [x] **1.10. Коммит/пуш** (сделано); деплой/health — при живом переключении утром.

**Критерии приёмки:** контрактные тесты зелёные; RLS-тест конкурентных
пользователей зелёный; под нагрузочным прогоном (например, `hey`/`ab`, 200
конкурентных «добавить заказ» с мока OSRM с задержкой 3 с) сервер не растит
потоки/память безгранично и отвечает; `/api/health` живой после деплоя.

## Фаза 2. Адреса слоями: Nominatim → DaData → pg_trgm → выбор (п.1)

**Цель:** неточный ввод адреса перестаёт быть тупиком, но НИКОГДА не
подставляется молча — при любой неуверенности юзер выбирает из 2–3 кандидатов
одним тапом («убирай труд, но не убирай контроль»).

### Схема

```
Ввод → Nominatim (есть) → уверенное совпадение? → готово
     → DaData «Подсказки» (через НАШ сервер-прокси) → кандидаты
     → нет сети/лимита → pg_trgm по таблице улиц из OSM (локально в PG)
     → уверенности нет → 2–3 кандидата на выбор, подтверждение одним тапом
```

### Шаги

- [x] **2.1. Таблица улиц из OSM.** Таблица `osm_streets` (регион, город, улица,
      street_norm, центроид) + pg_trgm + GIN-индекс в `db.py`. Экстрактор на сервере 2 —
      `osm_streets_pbf_service` (osmium из той же `russia5.osm.pbf`, привязка улицы к
      ближайшему нас. пункту по сетке, CSV.gz; CLI `python -m ...`). Загрузчик на
      сервере 1 — `osm_streets_import_service` (`import_from_url` качает артефакт как
      парковка). Осталось (сервер 2 / `main`): раздача `osm_streets.csv.gz` на :5001 и
      строка в `/opt/osrm/refresh.sh` — см. Журнал.
- [x] **2.2. Слой pg_trgm на сервере.** `OsmStreetRepository.search`: `similarity()` +
      оператор `%` по GIN-индексу, порог 0.3, фолбэк на ILIKE если pg_trgm нет.
      Всё квалифицировано схемой `public` (иначе не найдётся вне search_path).
- [x] **2.3. Прокси DaData.** `POST /api/address/suggest` (`routers/address.py`);
      клиент `dadata_service` (ключ из env `DADATA_API_KEY`, сигнатура сверена с
      dadata.ru); суточный лимит на пользователя — таблица `dadata_usage` +
      `DadataUsageRepository`; 403/сеть → тихий переход к pg_trgm. (Кеш подсказок —
      через существующий `address_cache` в Nominatim-слое; отдельный кеш DaData не
      делаем: сверка оферты показала, что достаточно rate-limit + офлайн-слоя.)
- [x] **2.4. Оркестратор слоёв** — `address_suggest_service.suggest`: координаты →
      Nominatim (уверенно при importance ≥ 0.5 и наличии дома) → DaData → pg_trgm →
      слабый Nominatim в кандидаты. Ответ всегда `{"resolved":…}` или
      `{"candidates":[0..3]}` — молча ничего не подставляется.
- [x] **2.5. Android: выбор кандидата.** Клиент `HomeVisitRepository.suggestAddress`
      зовёт `/api/address/suggest` перед расчётом; `resolved` → считаем сразу по его
      координатам; `candidates` (≥2) → компактный лист `AddressCandidatesList` в форме
      «Оценить заказ», тап = докрутка расчёта по координатам варианта (уходят как
      ручная точка `lat/lon` в `/api/visits/candidate` → `manual_geocoding_result`).
      Ручной маршрут подсказки не трогает. Компиляцию проверяет CI-сборка APK.
- [x] **2.6. Тесты:** опечатка («Ленена» → Ленина через pg_trgm/DaData), офлайн-путь
      pg_trgm при 403 DaData, исчерпание суточного лимита, счётчик, эндпоинт сквозь
      HTTP — `test_osm_streets.py` (10) + `test_address_suggest.py` (11), зелёные.
      Раскладку (qwerty→йцукен) исправляет сама DaData на своей стороне — проверяется
      уже при живом ключе, юнит-тестом мока не покрывается осмысленно.
- [x] **2.7. Коммит/пуш/APK.** Бэкенд и Android закоммичены и запушены; «Backend
      tests» и «Build Android APK» в CI зелёные; APK выложен на сайт
      (`vizitorkrut.ru/vizitorkrut.apk`, sha256 сверен с артефактом байт-в-байт).
      **Бэкенд-активация Фазы 2 едет с утренним переключением на FastAPI** (отложено
      в Фазе 1 под надзор Джавада): деплой `deploy_fastapi.sh` теперь сам мигрирует
      схему (`python -m app.db`, шаг 1b) — новые таблицы создадутся при переключении.
      Остаётся для полного офлайн-слоя (сервер 2 / `main`): наполнить `osm_streets`
      — прогнать `python -m app.services.osm_streets_pbf_service <pbf> osm_streets.csv.gz`,
      раздать файл на :5001, строкой в `/opt/osrm/refresh.sh` подхватывать импорт
      (сервер 1: `osm_streets_import_service.import_from_url`). Ключ DaData уже в env
      сервера (см. память). До наполнения `osm_streets` офлайн-слой пуст, но DaData и
      Nominatim работают, а APK деградирует мягко (при 404 подсказок — обычный путь).

**Критерии приёмки:** адрес с опечаткой находит правильную улицу через DaData;
при выключенном DaData (нет ключа/лимит) pg_trgm предлагает кандидатов; ни в
одном сценарии неуверенный адрес не подставляется без подтверждения тапом.

**Статус приёмки:** логика покрыта тестами (opечатка→pg_trgm/DaData, 403→pg_trgm,
молчаливой подстановки нет). Живая проверка на боевом — после утреннего FastAPI-
переключения и наполнения `osm_streets`.

## Фаза 3. Расчёты на телефон + батчинг GPS (п.4)

**Цель:** вердикт «стоит ли ехать» — мгновенный и офлайн; TSP уходит с сервера;
GPS шлётся пачками. Раздел труда: **сервер — карты (матрица), телефон —
арифметика и комбинаторика. Формула — на телефон, коэффициенты — с сервера.**

### Анти-разъезд логики (главный риск фазы)

- [ ] **3.1. Золотые векторы.** Из Python-тестов `profitability_service`
      генерируются JSON-фикстуры (входы → ожидаемые выходы). Kotlin-калькулятор
      обязан проходить ТО ЖЕ САМОЕ до копейки. Фикстуры — единый файл в репо,
      CI Android гоняет их в unit-тестах. «На телефоне 1500 ₽, на сервере
      1480 ₽» — покрытый тестом невозможный сценарий.

### Шаги

- [ ] **3.2. Эндпоинт матрицы.** `POST /api/route/matrix`: сервер зовёт OSRM
      (кеш из Ф1.5), отдаёт матрицу расстояний/времени (20×20 ≈ 3 КБ) + версию
      снапшота настроек. Fallback по прямой — тоже сервером (флаг в ответе).
- [ ] **3.3. Kotlin: ProfitabilityCalculator.** Перенос формулы выгодности
      (доход − км×топливо×(1+амортизация) − время, пороги, вердикт go/edge/skip,
      балл 0–100) в отдельный модуль. Коэффициенты — из кешированных
      `/api/settings` (кеш в Room уже есть). Проходит золотые векторы (3.1).
- [ ] **3.4. Kotlin: TSP.** nearest-neighbor + 2-opt (и полный перебор ≤8 точек,
      якоря фикс-времени — как на сервере) на матрице из 3.2. Считается по
      событию (добавление/завершение заказа), НЕ фоновым циклом — батарея
      водителя священна.
- [ ] **3.5. Пересчёт при завершении визита** — локально по кешированной матрице,
      без сети. Матрица инвалидируется при добавлении/удалении адреса.
- [ ] **3.6. Сервер — источник правды.** При синке сервер пересчитывает день;
      расхождение с клиентским расчётом >1 ₽ пишется в лог как баг (с версиями
      настроек) — сигнал разъезда формул.
- [ ] **3.7. Батчинг GPS.** Точки копятся в Room-очереди, отправка пачкой раз в
      5 минут или при 10 точках (образец батчинга — телеметрия вождения,
      `sendDrivingAggregateSync`). Новый `POST /api/location/batch`; сервер
      обрабатывает точки по очереди тем же конвейером (визиты, парковка —
      порядок и `segment_index` сохранить). Старый одиночный `/location`
      остаётся для старых APK. Немедленная отправка — только последняя точка
      пачки, если по ней нужен живой ответ (`ready_to_complete`, парковка):
      проверить и сохранить текущую механику алертов.
- [ ] **3.8. Дневные суммы для UI** — из Room локально (визиты дня уже там),
      сервер для этого не дёргается.
- [ ] **3.9. Тесты офлайна:** режим полёта → добавление заказа по кешированной
      матрице даёт вердикт; завершение визита пересчитывает маршрут; при
      возврате сети всё синкается без дублей.
- [ ] **3.10. Коммит/пуш/деплой; APK на сайт.**

**Чего в этой фазе НЕ делать (перегибы):** историю/статистику/индексы на телефон
не тащить (остаются серверу — им нужна полная история); карты на телефон не
кешировать; постоянный фоновый CPU не заводить.

**Критерии приёмки:** вердикт в самолётном режиме < 100 мс; золотые векторы
зелёные в обоих CI; число запросов GPS с устройства за час смены упало ~в 5 раз;
лог расхождений пуст на реальном дне работы.

## Фаза 4. Окно прибытия ±2 ч и цена фикс-времени (п.7)

**Цель:** честное окно прибытия по цепочке всего дня (не фейково-точный ETA) и
явная цена услуги «клиенту удобно в 15:00» в ₽/час — душа продукта «стоит ли?».

### Шаги

- [ ] **4.1. ETA-цепочка.** Расчёт по порядку дня: время дороги (личная поправка —
      уже есть) + длительности визитов (личная история). Неопределённость
      **накапливается** к концу дня — окно каждого следующего адреса шире.
      Начальная ширина и рост — из личной дисперсии юзера; дефолт ±1 ч у первого
      визита, до ±2 ч дальше. Обещанное окно не «прыгает» при мелких сдвигах —
      пересчёт с гистерезисом.
- [ ] **4.2. UI окна.** На карточке визита: «примерно 14:00–16:00» — словами,
      которые можно сказать клиенту по телефону. Показ и в Ленте (по цепочке).
- [ ] **4.3. Цена фикс-времени.** Якорная оптимизация уже есть
      (`_order_around_anchors`). Добавить сравнение: доход/час дня С якорем vs
      БЕЗ. Дельту показать при установке жёсткого времени: «фиксация 15:00
      стоит ~300 ₽/час сегодня». Не морализируя — решает юзер.
- [ ] **4.4. Подсказка наценки.** Если дельта существенна — ненавязчивая строка:
      «можно предложить клиенту наценку за точное время: ~N ₽» (N — из дельты
      и длительности). Осознанность → деньги.
- [ ] **4.5. Расчёты — на телефоне** (по матрице из Ф3), сервер — при синке.
      Пороги/коэффициенты — серверные настройки.
- [ ] **4.6. Тесты:** окно ширится к концу дня; якорь меняет дельту; лексика UI —
      по нейтральному словарю проекта.
- [ ] **4.7. Коммит/пуш/деплой; APK на сайт.**

## Фаза 9. Вердикт говорит всю правду: за круг + парковка (PAIN §1–2, §8)

**Боль (массовая):** порожняк — «приезжаешь на заказ, а там в жопу мира,
откуда будешь выбираться час»; таксисты жмут «На месте» за 300 метров и
отключают интернет, только чтобы узнать правду о заказе. Плюс парковка у
точки, которую никто не закладывает в цену.

**Суть:** вердикт считает полный круг (туда + маршрут дня + возврат) и деньги
парковки — и показывает разложение честно: «туда 12 км + обратно порожняком
12 км → реально 31 ₽/км, ниже твоей нормы».

### Шаги

- [ ] **9.1. Аудит текущей формулы: где теряется возврат.** Проверить по коду
      (`profitability_service.calculate_candidate_impact` и Kotlin-калькулятор
      после Ф3): входит ли плечо «последний визит → финиш дня» в маржинальную
      стоимость кандидата. День с геокодированным финишем уже должен его
      содержать (маршрут строится до финиша) — подтвердить тестом; если финиш
      дня не задан, возврат считать к точке старта дня. Результат аудита —
      в Журнал решений.
- [ ] **9.2. Возврат — всегда в расчёте.** Для всех режимов оценки кандидата
      обратное плечо (до финиша/старта дня) входит в марж. км и минуты. Прогноз
      обратных заказов НЕ делаем (данных нет — честный километраж вместо
      гадания; идея на цикл 3, отметить в PAIN_RESEARCH).
- [ ] **9.3. Разложение круга в UI вердикта.** Под кольцом выгодности —
      раскрываемая строка: «подъезд X км · визит · возврат Y км → всего Z км,
      N ₽/км против твоей нормы M ₽/км». Языком фольклора работников: ₽/км и
      ₽/мин крупно. Себестоимость км уже есть (`vehicle_km_cost`).
- [ ] **9.4. Парковка — в деньгах вердикта.** Проверка зоны на оценке уже
      дёргается (см. CLAUDE.md) — добавить ДЕНЬГИ: тариф зоны
      (`parking_street_prices`/`parking_tariffs`) × ожидаемая длительность
      визита (личная история длительностей). Правило памяти проекта «про цену
      не врать»: показывать вилку («парковка ~100–200 ₽»), в расчёт включать
      нижнюю границу; строка в разложении: «точка в платной зоне: +~150 ₽».
      Для пеших/вело-профилей — не показывать.
- [ ] **9.5. Золотые векторы расширить** (Python + Kotlin, единые фикстуры,
      как в Ф3.1): кейсы «финиш задан/не задан», «зона платная/нет»,
      «возврат длиннее подъезда» (загород). Формула не имеет права разъехаться.
- [ ] **9.6. Тесты:** заказ «в жопу мира» с пустым возвратом даёт skip при
      хорошем чеке в одну сторону; тот же заказ при финише дня рядом с точкой Б
      даёт go (возврат почти нулевой); платная зона снижает балл; офлайн-режим
      (матрица кеширована) показывает то же разложение.
- [ ] **9.7. Коммит/пуш/деплой; APK на сайт.**

**Критерии приёмки:** в разложении вердикта видны все плечи круга и парковка;
заказ, выгодный «в одну сторону» и убыточный «за круг», получает честный skip
с объяснением; золотые векторы зелёные в обоих CI; расхождений клиент/сервер
в логе нет.

## Фаза 10. Живая экономика смены: безубыточность, алерт «в минус», простой (PAIN §1, §7, §9)

**Боль (массовая):** «17 540 ₽ за день → 500 ₽ чистыми», «баланс за рабочий
день — минус 50 рублей» — и человек узнаёт это вечером. Арендник живёт на
беговой дорожке «800 ₽/час обязательных трат» и не видит, когда день отбит.
Ожидание у подъезда молча съедает выгодность заказа.

**Суть:** смена получает живой финансовый пульс: когда отбиты обязательные
расходы, не уходит ли день в минус, что делает с заказом ожидание — всё в
реальном времени, на телефоне (после Ф3 данные дня уже локальные).

### Шаги

- [ ] **10.1. Настройка фикс-расходов смены.** Новые ключи в каталоге настроек
      (`SETTINGS_CATALOG`, секция «Экономика»): «аренда авто, ₽/смена»
      (опционально, по умолчанию 0) и «прочие обязательные расходы смены, ₽»
      (мойка, связь). Обе редактируются на телефоне, синкаются как обычные
      настройки. Никакого обязательного онбординга — у кого свой авто, тот
      не заполняет и фича молчит про аренду.
- [ ] **10.2. Точка безубыточности.** Расчёт на телефоне из локальных данных:
      накопленный чистый доход дня (доходы завершённых визитов минус топливо
      по факту км, износ, фикс-расходы пропорционально) пересекает ноль →
      момент «смена отбита». UI: строка прогресса на Штурвале и в Ленте:
      до — «до безубыточности осталось N ₽», после — «смена отбита в 16:40,
      дальше работаешь на себя». Сервер дублирует расчёт при синке
      (источник правды, лог расхождений — как Ф3.6).
- [ ] **10.3. Алерт «смена уходит в минус».** Локальное уведомление (канал
      приложения, тон заботы, нейтральная лексика) по двум правилам:
      (а) три завершённых заказа подряд с чистой ₽/км ниже себестоимости;
      (б) середина смены пройдена, накопленный чистый — отрицательный.
      Не чаще одного раза за смену на каждое правило; текст объясняет причину
      цифрами и НЕ приказывает («последние 3 заказа ниже твоей себестоимости
      33 ₽/км — загляни в Ленту»). Отключаемо в настройках уведомлений.
- [ ] **10.4. Таймер простоя на точке.** Сигнал «прибыл, визит не завершён» уже
      есть (GPS + `ready_to_complete`). На карточке активного визита — живой
      пересчёт: «ожидание 18 мин: заказ уже 210 ₽/час вместо 350». Обновление
      раз в минуту, только при открытом экране (батарея священна). При падении
      ниже личной нормы — мягкая строка «ожидание съело выгоду ниже твоей
      нормы», без давления и без автодействий.
- [ ] **10.5. Тесты:** аренда не задана → блок безубыточности не показывает
      аренду и стартует с топлива; день в минус → алерт ровно один; ожидание
      двигает ₽/час монотонно вниз; всё работает офлайн; лексика — по
      нейтральному словарю (никакой «усталости»).
- [ ] **10.6. Коммит/пуш/деплой; APK на сайт.**

**Критерии приёмки:** арендник видит момент «смена отбита» без единого
лишнего тапа; день, идущий в минус, «кричит» днём, а не вечером; на визите
с ожиданием цифра ₽/час тает на глазах; при выключенных уведомлениях —
тишина, при пустых настройках аренды — ничего лишнего.

## Фаза 11. Минимальный чек по адресу + цена отмены (PAIN §4, §6)

**Боль (массовая/регулярная):** «никто не поедет через весь город менять
смеситель за 800 рублей» (Фонтанка, 04.2026) — а минимальную цену выезда все
считают на глазок во время звонка клиента. И: «клиент отменил, когда я уже
ехал» — убыток, который никто не фиксирует и не копит.

**Суть:** два инструмента вокруг заказа. ДО: «назови цену за 10 секунд» —
по адресу мгновенно посчитать, ниже какой суммы выезд убыточен. ПОСЛЕ:
кнопка «клиент отменил» фиксирует реальные потери и копит статистику,
из которой рождается совет о предоплате.

### Шаги

- [ ] **11.1. Экран «Быстрая оценка» (минимальный чек).** Вход со Штурвала,
      работает и вне смены. Поток: вставил/надиктовал адрес → слоёный геокодинг
      из Ф2 (подсказки DaData ускоряют до «10 секунд») → расчёт от текущей
      позиции (или дома, если GPS выключен): дорога туда-обратно ×
      себестоимость км + время дороги × личная норма ₽/час + парковка (Ф9.4)
      → крупно: «минимальный чек: 2 400 ₽» + строка «час работы на месте =
      +N ₽». Кнопка «в заказ» переносит адрес в создание визита. Офлайн:
      расчёт по прямой с честной пометкой «оценка».
- [ ] **11.2. Источник заказа (опциональное поле).** При создании визита —
      необязательный выбор: Профи / Авито / YouDo / сарафан / другое +
      «цена отклика, ₽» (для платных лидов). Ноль обязательности: пропустил —
      ничего не спрашиваем. Цена отклика входит в расчёт выгодности заказа.
- [ ] **11.3. Кнопка «Клиент отменил».** На принятом/активном визите. По GPS
      уже проеханного сегмента фиксируем факт: км, минуты, деньги (км ×
      себестоимость + время × норма) → визит получает статус «отменён в пути»
      с суммой потерь. Ничего не спрашиваем в моменте — одна кнопка, человек
      за рулём.
- [ ] **11.4. Статистика отмен и пустых лидов.** В недельной/месячной сводке:
      «отмены съели X ₽ (Y поездок), отклики без заказа — Z ₽» с разбивкой по
      источникам («Профи: 3 400 ₽ откликов → 1 заказ»). При доле потерь выше
      порога (серверная настройка) — деликатный совет: «на адресах дальше N км
      стоит брать предоплату за выезд». Совет, не приказ.
- [ ] **11.5. Тесты:** быстрая оценка без смены и без сети; отмена в пути
      пишет ровно фактический км/мин (не плановый маршрут); отменённые визиты
      не портят личные нормы длительности/дохода и индексы; сводка складывает
      потери корректно; поле источника пустое — статистика без разбивки.
- [ ] **11.6. Коммит/пуш/деплой; APK на сайт.**

**Критерии приёмки:** от звонка клиента до цифры минимального чека ≤ 10 секунд
(адрес с подсказкой + одна кнопка); отмена в пути фиксируется одним тапом и
видна в сводке с деньгами; накопленная статистика по источникам отвечает на
вопрос «окупает ли себя Профи» реальными цифрами юзера.
