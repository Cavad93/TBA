# Визиторкрут — мастер-план доработок (июль 2026)

> Единый поэтапный план по семи направлениям. Исполняется **автономно до полного
> завершения**: работа не считается законченной, пока каждый чекбокс не отмечен
> и критерии приёмки каждой фазы не выполнены. Чекбоксы отмечаются в этом файле
> по мере выполнения, решения по ходу — в «Журнале решений» внизу.

## Правила автономного исполнения (обязательны)

1. **Не останавливаться до 100%.** Проект завершён только когда все фазы закрыты,
   все критерии приёмки проверены на живом сервере/устройстве, чекбоксы отмечены.
2. **Блокер ≠ остановка.** Обнаружил проблему → изучить код и интернет → выбрать
   лучшее решение → применить → записать в Журнал решений. Пользователя дёргать
   только по пунктам из раздела «Действия Джавада» и по деньгам.
3. **Каждая фаза заканчивается выкаткой:** коммит + `git push` в
   `claude/android-migration-plan-review-gbiba7`; при серверных изменениях —
   деплой на сервер 1 и проверка `/api/health`; при клиентских — дождаться APK
   в GitHub Actions и выложить на сайт (порядок — в CLAUDE.md).
4. **Контракты API не ломать молча.** Android-клиенты в поле обновляются не сразу:
   старые эндпоинты живут, пока новая версия APK не станет обязательной.
5. **Модульность:** новые роутеры/сервисы/экраны — отдельными файлами,
   ориентир ~500 строк.
6. **Чувствительные места** (152-ФЗ, платёжные/партнёрские интеграции, сигнатуры
   внешних API) перед реализацией сверять с актуальными источниками в сети.

## Фаза 1. Фундамент: переезд на FastAPI/uvicorn, пул, async (п.3)

**Цель:** сервер выдерживает 1000+ пользователей: пул соединений PG, отсутствие
неограниченных потоков, OSRM не держит воркеры, тяжёлый TSP не блокирует API.

**Решение (выбрано Джавадом):** полный переезд с `ThreadingHTTPServer`
(`app/location_api.py`) на FastAPI + uvicorn. Меры против регрессий — контрактные
тесты до/после и параллельный прогон на тестовом порту.

### Шаги

- [x] **1.1. Контрактные тесты текущего API.** Снять «золотые» запросы/ответы всех
      эндпоинтов `location_api.py` (GET/POST/DELETE, авторизация, коды ошибок,
      формат тел). Это страховка всего переезда — делается ДО любого переноса.
- [x] **1.2. Каркас FastAPI.** `app/api/` с роутерами по доменам (auth, day, visits,
      location, settings, shift, stats, health...) — один домен = один файл.
      Pydantic-модели тел. Существующие сервисы вызываются как есть.
- [x] **1.3. Пул соединений с RLS.** `psycopg_pool.AsyncConnectionPool`.
      FastAPI-dependency: взять соединение из пула → `set_config('app.user_id', …)`
      → отдать хендлеру → при возврате в пул сбросить GUC (`RESET`). Тест: два
      конкурентных запроса разных пользователей не видят чужие строки (RLS).
      Комментарий в `database.py` про запрет session-пулера относится к внешнему
      pgbouncer — внутренний пул с явным set/reset безопасен, но это проверяется
      тестом, а не верой.
- [x] **1.4. OSRM async.** `httpx.AsyncClient`, таймауты как сейчас, +1 retry на
      сетевые ошибки (не на 400!). Сохранить святое: `NoSegment` с HTTP 400 →
      `OutsideCoverageError` → человеку словами; `raise_for_status` только ≥500.
- [x] **1.5. Кеш матриц OSRM.** In-memory TTL-кеш (ключ — округлённые координаты
      точек + профиль), инвалидация по TTL ~10 мин. Экономит и OSRM, и латентность.
- [x] **1.6. CPU-bound в executor.** Пока TSP не переехал на телефон (Ф3) — гонять
      `optimization_service._best_order` через `run_in_executor(ProcessPoolExecutor)`,
      чтобы event loop не вставал.
- [x] **1.7. Лимиты.** uvicorn: `--limit-concurrency`, воркеры по ядрам сервера 1;
      серверные таймауты; лимит размера тела запроса.
- [x] **1.8. Прогон контрактных тестов** против FastAPI на тестовом порту — ответы
      байт-в-байт совпадают с золотыми (кроме заголовков сервера).
- [x] **1.9. Деплой ВЫПОЛНЕН (15.07.2026).** Живое переключение боевого трафика на
      FastAPI/uvicorn проведено по явной авторизации Джавада (`deploy/deploy_fastapi.sh`:
      миграция схемы → параллельный прогон на 8089 → смоук → смена systemd-юнита →
      рестарт). Служба `homevisit-api` теперь `uvicorn app.asgi:app` (2 воркера,
      `--limit-concurrency 64`), `/api/health` живой, `NRestarts=0`, ошибок в логах нет.
      Смоук маршрутизации: `/location`, `/api/home/shift/settings`, `/api/address/suggest`
      отвечают 401 без токена (зарегистрированы, авторизация работает). Откат — одной
      строкой (закомментированный `ExecStart` на `app.main` в юните).
- [x] **1.10. Коммит/пуш/деплой/health — выполнено.**

**Критерии приёмки:** контрактные тесты зелёные; RLS-тест конкурентных
пользователей зелёный; под нагрузочным прогоном (например, `hey`/`ab`, 200
конкурентных «добавить заказ» с мока OSRM с задержкой 3 с) сервер не растит
потоки/память безгранично и отвечает; `/api/health` живой после деплоя.

## Фаза 2. Адреса слоями: Nominatim → DaData → pg_trgm → выбор (п.1)

**Цель:** неточный ввод адреса перестаёт быть тупиком, но НИКОГДА не
подставляется молча — при любой неуверенности юзер выбирает из 2–3 кандидатов
одним тапом («убирай труд, но не убирай контроль»).

### Схема

```
Ввод → Nominatim (есть) → уверенное совпадение? → готово
     → DaData «Подсказки» (через НАШ сервер-прокси) → кандидаты
     → нет сети/лимита → pg_trgm по таблице улиц из OSM (локально в PG)
     → уверенности нет → 2–3 кандидата на выбор, подтверждение одним тапом
```

### Шаги

- [x] **2.1. Таблица улиц из OSM.** Таблица `osm_streets` (регион, город, улица,
      street_norm, центроид) + pg_trgm + GIN-индекс в `db.py`. Экстрактор на сервере 2 —
      `osm_streets_pbf_service` (osmium из той же `russia5.osm.pbf`, привязка улицы к
      ближайшему нас. пункту по сетке, CSV.gz; CLI `python -m ...`). Загрузчик на
      сервере 1 — `osm_streets_import_service` (`import_from_url` качает артефакт как
      парковка). Осталось (сервер 2 / `main`): раздача `osm_streets.csv.gz` на :5001 и
      строка в `/opt/osrm/refresh.sh` — см. Журнал.
- [x] **2.2. Слой pg_trgm на сервере.** `OsmStreetRepository.search`: `similarity()` +
      оператор `%` по GIN-индексу, порог 0.3, фолбэк на ILIKE если pg_trgm нет.
      Всё квалифицировано схемой `public` (иначе не найдётся вне search_path).
- [x] **2.3. Прокси DaData.** `POST /api/address/suggest` (`routers/address.py`);
      клиент `dadata_service` (ключ из env `DADATA_API_KEY`, сигнатура сверена с
      dadata.ru); суточный лимит на пользователя — таблица `dadata_usage` +
      `DadataUsageRepository`; 403/сеть → тихий переход к pg_trgm. (Кеш подсказок —
      через существующий `address_cache` в Nominatim-слое; отдельный кеш DaData не
      делаем: сверка оферты показала, что достаточно rate-limit + офлайн-слоя.)
- [x] **2.4. Оркестратор слоёв** — `address_suggest_service.suggest`: координаты →
      Nominatim (уверенно при importance ≥ 0.5 и наличии дома) → DaData → pg_trgm →
      слабый Nominatim в кандидаты. Ответ всегда `{"resolved":…}` или
      `{"candidates":[0..3]}` — молча ничего не подставляется.
- [x] **2.5. Android: выбор кандидата.** Клиент `HomeVisitRepository.suggestAddress`
      зовёт `/api/address/suggest` перед расчётом; `resolved` → считаем сразу по его
      координатам; `candidates` (≥2) → компактный лист `AddressCandidatesList` в форме
      «Оценить заказ», тап = докрутка расчёта по координатам варианта (уходят как
      ручная точка `lat/lon` в `/api/visits/candidate` → `manual_geocoding_result`).
      Ручной маршрут подсказки не трогает. Компиляцию проверяет CI-сборка APK.
- [x] **2.6. Тесты:** опечатка («Ленена» → Ленина через pg_trgm/DaData), офлайн-путь
      pg_trgm при 403 DaData, исчерпание суточного лимита, счётчик, эндпоинт сквозь
      HTTP — `test_osm_streets.py` (10) + `test_address_suggest.py` (11), зелёные.
      Раскладку (qwerty→йцукен) исправляет сама DaData на своей стороне — проверяется
      уже при живом ключе, юнит-тестом мока не покрывается осмысленно.
- [x] **2.7. Коммит/пуш/APK.** Бэкенд и Android закоммичены и запушены; «Backend
      tests» и «Build Android APK» в CI зелёные; APK выложен на сайт
      (`vizitorkrut.ru/vizitorkrut.apk`, sha256 сверен с артефактом байт-в-байт).
      **Бэкенд Фазы 2 ПОЛНОСТЬЮ АКТИВЕН (15.07.2026):** живое переключение на
      FastAPI/uvicorn (см. 1.9); ключи DaData в env сервера (проверено — 3 подсказки с
      координатами); `osm_streets` наполнена — 462 513 улиц из `russia5.osm.pbf`
      (экстрактор на сервере 2 → `osm_streets.csv.gz` на :5001 → импорт на сервер 1).
      Все три слоя работают на проде. Осталось из инфраструктуры: строку импорта улиц
      добавить в `/opt/osrm/refresh.sh` (в `main`), чтобы обновлялось раз в 2 месяца
      вместе с зонами парковки — сейчас наполнено разово вручную.

**Критерии приёмки:** адрес с опечаткой находит правильную улицу через DaData;
при выключенном DaData (нет ключа/лимит) pg_trgm предлагает кандидатов; ни в
одном сценарии неуверенный адрес не подставляется без подтверждения тапом.

**Статус приёмки:** логика покрыта тестами (opечатка→pg_trgm/DaData, 403→pg_trgm,
молчаливой подстановки нет). Живая проверка на боевом — после утреннего FastAPI-
переключения и наполнения `osm_streets`.

## Фаза 3. Расчёты на телефон + батчинг GPS (п.4)

**Цель:** вердикт «стоит ли ехать» — мгновенный и офлайн; TSP уходит с сервера;
GPS шлётся пачками. Раздел труда: **сервер — карты (матрица), телефон —
арифметика и комбинаторика. Формула — на телефон, коэффициенты — с сервера.**

### Анти-разъезд логики (главный риск фазы)

- [x] **3.1. Золотые векторы.** Из Python-тестов `profitability_service`
      генерируются JSON-фикстуры (входы → ожидаемые выходы). Kotlin-калькулятор
      обязан проходить ТО ЖЕ САМОЕ до копейки. Фикстуры — единый файл в репо,
      CI Android гоняет их в unit-тестах. «На телефоне 1500 ₽, на сервере
      1480 ₽» — покрытый тестом невозможный сценарий. (Готово: `candidate_vectors.json`
      + `scripts/gen_golden_vectors.py`; аналогично для маршрута — `route_vectors.json`
      + `scripts/gen_route_vectors.py`, см. 3.4.)

### Шаги

- [x] **3.2. Эндпоинт матрицы.** `POST /api/route/matrix`: сервер зовёт OSRM
      (кеш из Ф1.5), отдаёт матрицу расстояний/времени (20×20 ≈ 3 КБ) + версию
      снапшота настроек. Fallback по прямой — тоже сервером (флаг в ответе).
      (Готово: `matrix_service.py` + `routers/route.py` + `test_matrix_service.py`.)
- [x] **3.3. Kotlin: ProfitabilityCalculator.** Перенос формулы выгодности
      (доход − км×топливо×(1+амортизация) − время, пороги, вердикт go/edge/skip,
      балл 0–100) в отдельный модуль. Коэффициенты — из кешированных
      `/api/settings` (кеш в Room уже есть). Проходит золотые векторы (3.1).
- [~] **3.4. Kotlin: TSP.** nearest-neighbor + 2-opt (и полный перебор ≤8 точек,
      якоря фикс-времени — как на сервере) на матрице из 3.2. Считается по
      событию (добавление/завершение заказа), НЕ фоновым циклом — батарея
      водителя священна. **Ядро готово:** `RouteOptimizer.kt` + `route_vectors.json`
      + `RouteOptimizerTest.kt`, перенос сверен Python-зеркалом (все 5 векторов).
      Клиентский слой матрицы (`fetchRouteMatrix` + `MatrixSnapshot` + кеш) готов.
      **Композиция офлайн-вердикта готова и покрыта CI-тестами:** `OfflineVerdict.kt`
      (склейка candidateExtra + дневная арифметика + ProfitabilityCalculator) +
      `OfflineCandidateEstimator.kt` (достройка плеч нового адреса по прямой —
      точь-в-точь серверный `get_estimated_distance_matrix`) + `OfflineVerdictTest`/
      `OfflineCandidateEstimatorTest` (зелёные в CI). **Офлайн-вердикт вызывается из
      ViewModel как fallback (готово):** новый `GET /api/route/matrix/day` собирает точки
      дня (старт/принятые/финиш) с КООРДИНАТАМИ + матрицу OSRM + доходы (клиент координат
      заказов не хранит); `warmDayMatrix` кеширует их при обновлении Ленты;
      `OfflineEstimateMapper` строит вердикт из кеша; `runCandidate` при `network_error`
      с координатами кандидата отдаёт мгновенную офлайн-оценку. Тесты: `test_day_matrix.py`
      (2, OSRM), `OfflineEstimateMapperTest` (JVM), инструментальные — эмулятор/Test Lab.
- [~] **3.5. Пересчёт при завершении визита** — локально по кешированной матрице,
      без сети. Матрица инвалидируется при добавлении/удалении адреса.
      Ядро пересчёта — `OfflineVerdict`/`RouteOptimizer` по кеш-матрице (готово,
      CI-тесты). **Инвалидация кеша — через прогрев (готово):** `completeCurrentVisit`/
      принятие/удаление зовут `refreshRouteInternal` → `warmDayMatrix`, поэтому кеш
      матрицы дня всегда отражает текущий набор точек; офлайн-вердикт считает по свежему.
- [x] **3.6. Сервер — источник правды.** При синке сервер пересчитывает день;
      расхождение с клиентским расчётом >1 ₽ пишется в лог как баг (с версиями

## Фаза 9. Вердикт говорит всю правду: за круг + парковка (PAIN §1–2, §8)

**Боль (массовая):** порожняк — «приезжаешь на заказ, а там в жопу мира,

## Фаза 10. Живая экономика смены: безубыточность, алерт «в минус», простой (PAIN §1, §7, §9)

## Фаза 11. Минимальный чек по адресу + личный режим + цена отмены (PAIN §4, §6)

**Боль (массовая/регулярная):** «никто не поедет через весь город менять
смеситель за 800 рублей» (Фонтанка, 04.2026) — а минимальную цену выезда все
считают на глазок во время звонка клиента. И: «клиент отменил, когда я уже
ехал» — убыток, который никто не фиксирует и не копит.

**Суть:** два инструмента вокруг заказа. ДО: «назови цену за 10 секунд» —

кнопка «клиент отменил» фиксирует реальные потери и копит статистику,
из которой рождается совет о предоплате.

Плюс **личный режим** (решение Джавада 15.07.2026): тот же расчёт стоимости
поездки — для личных целей. НЕ развилка при регистрации (расфокус продукта),

# ЦИКЛ 3 — ввод адресов без труда (голос, список, скриншот)

> Решение Джавада 15.07.2026: полностью переработать ввод адресов — распознавание
> ошибок, голосовой ввод, загрузка списком, скриншоты. Всё на СВОИХ движках
> (никаких платных сторонних API); тяжёлые движки — на сервере 2 (16 ГБ RAM),
> сервер 1 не трогаем. Суть: вводить максимально просто и быстро, без лишних
> действий. Запуск после закрытия фазы 12.
>
> **Бюджет RAM сервера 2:** OSRM (3 инстанса) + ASR (~1–1,5 ГБ) + OCR по
> требованию (~1 ГБ) — помещается; на время пересборки графа (пик 14,4 ГБ раз
> в 2 месяца) вспомогательные движки останавливаются скриптом refresh.sh и
> поднимаются после — деградация: клиент в эти минуты живёт на системном
> распознавании и ручном вводе.
>
> **Приватность голоса и фото:** аудио и изображения нигде не сохраняются —
> транскрибируются в памяти и выбрасываются, в логи не пишутся. Канал
> сервер 1 ↔ сервер 2 для аудио/фото — только шифрованный (HTTPS через Caddy
