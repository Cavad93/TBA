## 1. Утверждённая сетка (финальная, версия Джавада)

| Тариф | Что даёт | Логика |
|---|---|---|
| **Free** | Честный вердикт по заказу **включая парковку и износ**; лента на полный день (**без лимита**, ручной порядок); безубыточность дня; **все остальные фичи**, кроме входящих в Standard/Pro | Правда + привычка + вирусный крючок; обещание бренда цело |
| **Standard 149 ₽/мес (1490 ₽/год)** | Авто-оптимизация маршрута дня; история/тренды маржи; синк между устройствами; заморозка на мёртвый сезон | Рычаг поверх правды + память |
| **Pro 249 ₽/мес (2490 ₽/год)** | Бенчмарк «ниже медианы города» (когда накопятся данные); автопарк-тариф для газелистов (несколько машин — ARPU ×5); симуляция/прогноз дохода | «Заработать больше», а не «стоит ли» |

**Дельты от вердикта совета (осознанные решения Джавада):**
- «Экспорт для „Мой налог“/кредита» **не включён** в Standard (совет предлагал).
  Остаётся кандидатом на добор ценности Standard — см. §6.8.
- Free сформулирован широко: «всё, кроме платного». Это правильная рамка: новая
  фича по умолчанию бесплатна, платность — явное решение.

**Принципы (из вердикта, обязательны при реализации):**
1. **Правда бесплатно, рычаг платно.** Ни одна цифра себестоимости (топливо, износ,
   парковка, налог) не прячется за paywall ни в каком виде. Никаких «серых замков
   с суммой» на компонентах правды — это тёмный паттерн (рецензия заседания 2).
2. **Сервер — источник правды по entitlements.** Клиент кеширует, сервер решает.
3. **Состав тарифов и цены — серверные настройки** (как `osago_partner_url`):
   меняются без релиза APK.
4. **Лимитов на объём нет вообще** (ни оценок/день, ни размера ленты) — ось
   объёма признана неверной.
5. **Годовой апфронт — главный платёжный инструмент**, пока рекуррент в РФ ненадёжен.
6. **Paywall не раньше «ага-момента»:** новому юзеру — 14 дней полного Standard
   (триал без карты), потом мягкая деградация до Free.

## 3. Архитектура entitlements (ядро всего)

### 3.1. Схема БД (новые таблицы в `app/db.py`, отдельная миграция)

```sql
CREATE TABLE IF NOT EXISTS subscriptions (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER NOT NULL,            -- владелец подписки
    plan TEXT NOT NULL,                  -- 'standard' | 'pro'
    status TEXT NOT NULL,                -- 'active' | 'frozen' | 'expired' | 'canceled'
    source TEXT NOT NULL,                -- 'yookassa' | 'rustore' | 'google' | 'manual' (промо/grandfather)
    period TEXT NOT NULL,                -- 'month' | 'year' | 'trial'
    started_at TEXT NOT NULL,
    expires_at TEXT NOT NULL,            -- продлевается при заморозке
    frozen_at TEXT, frozen_until TEXT,   -- заморозка сезона (§6.4)
    external_id TEXT,                    -- id платежа/подписки у провайдера
    created_at TEXT NOT NULL
);
CREATE TABLE IF NOT EXISTS payments (       -- журнал вебхуков, идемпотентность
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id INTEGER, provider TEXT NOT NULL,
    external_id TEXT NOT NULL UNIQUE,       -- id события у провайдера — защита от дублей
    amount NUMERIC, currency TEXT, plan TEXT, period TEXT,
    status TEXT NOT NULL,                    -- 'succeeded' | 'refunded' | ...
    raw_payload_json TEXT NOT NULL, created_at TEXT NOT NULL
);
CREATE TABLE IF NOT EXISTS fleet_members (   -- автопарк (§6.6)
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    owner_user_id INTEGER NOT NULL,          -- кто платит (Pro + места)
    member_user_id INTEGER NOT NULL UNIQUE,  -- водитель (одно членство)
    invited_code TEXT, consent_at TEXT,      -- согласие водителя видеть его сводку — явное
    created_at TEXT NOT NULL
);
```
RLS: `subscriptions`/`payments` — изоляция по `user_id`, как у остальных таблиц.
`fleet_members` — политика по `owner_user_id OR member_user_id` (обе стороны видят связь).

### 3.2. `entitlement_service.py` (новый, ~150 строк)

Единственная точка ответа на вопрос «что можно юзеру»:

```python
resolve(user_id) -> Entitlements(
    plan: 'free'|'standard'|'pro',
    features: {auto_optimize, history_trends, multi_device, freeze,
               benchmark, simulation, fleet},   # bool каждая
    expires_at, is_trial, frozen_until)
```

Правила: `pro` включает всё из `standard`; активная подписка = `status='active'
AND expires_at > now()`; заморозка → фичи выключены, но подписка не сгорает;
триал = запись `plan='standard', period='trial', source='manual'` при регистрации.
**Карта «фича → минимальный план» лежит в серверном конфиге** (`server_settings`,
ключ `plan_features_json`) — состав тарифа меняется без релиза и без миграции.
Кеш в памяти с TTL ~60 с (по образцу `server_settings._config`).

### 3.3. API (новый роутер `app/api/routers/billing.py`)

- `GET /api/billing/status` → entitlements + цены + текст тарифов (клиент рисует
  paywall по серверным данным — цены не зашиты в APK).
- `POST /api/billing/yookassa/webhook` → приём `payment.succeeded` (§5).
- `POST /api/billing/freeze` / `POST /api/billing/unfreeze` (§6.4).
- `POST /api/billing/fleet/invite`, `POST /api/billing/fleet/accept` (§6.6).
- Блок `entitlements` добавить в `/api/home` snapshot (`home_service.snapshot`) —
  клиент получает права вместе с главным экраном, без лишнего запроса.

### 3.4. Enforcement (сервер решает, клиент рисует)

| Фича | Точка проверки на сервере |
|---|---|
| Авто-оптимизация | эндпоинт/сервис оптимизации порядка (`optimization_service`) — при `free` возвращает порядок «как введено» + флаг `auto_optimize_locked` |
| История/тренды | новые эндпоинты `/api/stats/trends*` (§6.3) — 402-подобный ответ `{"locked": "standard"}`; бесплатный горизонт — текущий день + вчера |
| Мультиустройство | `auth_service.login`: при `free` — новая сессия **гасит старые** (`sessions.revoked_at`); при `standard+` — параллельные сессии живут |
| Заморозка | `billing.freeze` — только `standard+`, не в триале |
| Бенчмарк/симуляция/автопарк | соответствующие эндпоинты (§6.5–6.7) |

Клиент дублирует гейты по кешу entitlements (Room) для офлайна: кешированный
`plan` живёт 72 ч офлайна, потом мягкая деградация до Free (не наказывать за
плохую связь — grace, не отрезание).

**Чего сервер НЕ делает:** не выключает расчёт правды. Вердикт, парковка, износ,
безубыточность считаются всем и всегда — это не проверяется entitlement'ами вообще.

## 4. Этапы реализации (порядок обязателен)

### Этап T1. Ядро entitlements + триал (бэкенд, ~2-3 дня работы)
- [ ] Таблицы §3.1 + RLS + миграция (`python -m app.db` в деплой-скрипте).
- [ ] `entitlement_service.py` + конфиг `plan_features_json` в `server_settings`.
- [ ] `GET /api/billing/status`; блок `entitlements` в `/api/home`.
- [ ] Триал 14 дней Standard при регистрации (`auth_service._seed_user_settings` — рядом).
- [ ] Grandfathering-скрипт: всем существующим юзерам `manual`-подписка Standard
      до +90 дней (§7.1) — одноразово, в Журнал решений.
- [ ] Тесты: resolve по всем статусам; триал; истечение; pro⊃standard; RLS-изоляция.
- [ ] Деплой + health. **Ничего ещё не гейтится** — ядро едет вперёд тихо.

### Этап T2. Гейтинг Standard (бэкенд + клиент, после T1)
- [ ] Авто-оптимизация: сервер отдаёт `auto_optimize_locked` для free; порядок —
      «как введено» с ручной пересортировкой. Kotlin: кнопка «Оптимизировать
      маршрут» ведёт на paywall при locked; `RouteOptimizer` офлайн-ядро — гейт
      по кешу entitlements.
- [ ] Ручная пересортировка ленты (drag) — **обязательная пара к гейту**: Free
      без авто-порядка должен уметь ручной. Если drag'а нет в UI — сделать в этом
      же этапе, иначе Free сломан.
- [ ] Сессии: free = одна активная (логин гасит прежние с понятным текстом),
      standard+ = параллельные.
- [ ] Экран тарифов в приложении (цены/состав — из `/api/billing/status`).
- [ ] Тесты + CI-сборка APK + выкатка на сайт.

### Этап T3. Деньги: ЮKassa-канал сайта (§5) + аналитика
- [ ] Платёжная страница на сайте (Caddy-статика + `/api/billing/*`).
- [ ] Вебхук с идемпотентностью (`payments.external_id UNIQUE`).
- [ ] Событийная аналитика конверсии (без сторонних SDK): таблица `events`
      (paywall_shown / paywall_click / payment_succeeded) — без неё монетизацию
      не видно (предпосылка из вердикта).
- [ ] Первая проверка: годовой предзаказ 5-10 целевым мастерам по прямой ссылке.

### Этап T4. Standard-фичи «памяти»: история/тренды (§6.3), заморозка (§6.4)
### Этап T5. Pro-фичи: симуляция/прогноз (§6.5) → автопарк (§6.6) → бенчмарк (§6.7, последним — ему нужны данные)
### Этап T6. Сторы: RuStore Billing / Google Play Billing (§5.3) — только когда фаза 17 MASTER_PLAN дойдёт до сторов

Критерий каждого этапа — как в MASTER_PLAN: тесты зелёные в CI, деплой, health,
APK на сайт при клиентских изменениях.

## 5. Платёжный контур (чувствительное место — правило 6: сверить перед реализацией)

### 5.1. Канал «APK с сайта» — ЮKassa (первый и главный)
Схема без SDK в приложении (вердикт Экзекьютора, скорректированный рецензией):
кнопка в приложении/на сайте → браузер → платёжная страница ЮKassa → вебхук
`payment.succeeded` → `payments` (идемпотентно) → `subscriptions` → юзеру письмо.
Привязка платежа к аккаунту: платёжная ссылка генерируется сервером с `metadata.user_id`.

**Сверить при реализации (не принимать на веру):**
- Доступность ЮKassa для самозанятого vs требование ИП; альтернативы, если
  самозанятому недоступно полноценное API: CloudPayments, Продамус, Robokassa.
- Чек: при приёме через платёжного агрегатора на НПД чек в «Мой налог» бьёт сам
  Джавад на каждый платёж (лимит НПД 2,4 млн/год — далеко, но помнить).
- 54-ФЗ/возвраты: политика возврата на сайте обязательна для эквайринга.

### 5.2. Рекуррент — НЕ строить сейчас
Автосписания в РФ для этой аудитории хрупки (вердикт). Продажа — разовые периоды:
месяц/год, за 7 дней до конца — уведомление «продлить» (инфраструктура уведомлений
Ф7 MASTER_PLAN). Годовой — со скидкой ~2 мес (1490 = 149×10) как якорь.

### 5.3. Сторы (потом): RuStore Billing / Google Play Billing
В сторах внешняя ссылка на оплату = риск бана → там их встроенный биллинг.
`subscriptions.source` уже различает каналы. Верификация покупок — серверная
(вебхук/verify API стора). Делается в этапе T6, не раньше фазы 17.

## 6. Нереализованные фичи — как строить каждую

### 6.1. Free: парковка и износ в вердикте — НЕ строить здесь
Это Ф9.4 и Ф6/Ф9 MASTER_PLAN. Тарифное решение одно: **обе фичи Free навсегда**,
в `plan_features_json` их нет вообще (не «включены в free», а не являются фичами
тарифа). То же — безубыточность дня (Ф10.2).

### 6.2. Авто-оптимизация как платная фича (Standard)
**Есть:** весь движок (сервер `optimization_service._best_order`, клиент
`RouteOptimizer.kt` + матрицы Ф3). **Строится только гейт:**
- Серверный флаг в ответах, где строится порядок дня; при `free` — порядок ввода.
- Клиент: кнопка «⚡ Оптимизировать день» на ленте; locked → paywall с честным
  превью выгоды: «оптимизация сэкономила бы сегодня ~N км / ~M ₽» (посчитать МОЖНО
  бесплатно, показать маршрут — платно; это витрина ценности, не тёмный паттерн —
  цифра экономии и есть реклама рычага).
- Якоря фикс-времени (Ф4) уважаются в обоих режимах: ручной порядок не должен
  ломать обещанные окна прибытия.
- Тесты: free получает introduced-order + флаг; standard — оптимум; golden-векторы
  Ф3 не трогаются (движок не меняется, меняется доступ).

### 6.3. История и тренды маржи (Standard)
**Есть сырьё:** `DayMetricRepository.history`, `home_service._trends_payload`,
`stats_summary`. **Строить:**
- Бэкенд: `app/services/margin_trends_service.py` + роутер `/api/stats/trends`:
  недельные/месячные агрегаты — чистыми ₽/день, ₽/км, ₽/час, разбивка по
  источникам заказов (`order_source_label` уже в users/визитах), «системно
  убыточные зоны/клиенты» (агрегация по адресам с отрицательной маржой ≥3 раза).
- Free-горизонт: сегодня+вчера (текущие блоки `/api/home` не трогаются — что
  бесплатно сегодня, бесплатным и остаётся, см. §7.1); глубже — Standard.
- Клиент: экран «Тренды» (вкладка в статистике), графики по дизайн-системе.
- Ретеншн-примечание: GPS хранится 14 дней, но агрегаты дня (`day_metrics`) —
  долгоживущие; тренды строятся по агрегатам, НЕ по сырому GPS (152-ФЗ чисто).
- Тесты: агрегация недель/месяцев; free видит 2 дня; разбивка по источникам.

### 6.4. Заморозка на мёртвый сезон (Standard)
Простая и важная для удержания (сезонность газелиста):
- `POST /api/billing/freeze`: `status='frozen'`, `frozen_until` (макс. N дней в
  году — серверная настройка, старт: 90), `expires_at += длительность заморозки`.
- Во время заморозки Standard-фичи выключены (иначе заморозка = бесплатный тариф),
  подписка не сгорает. Unfreeze — в любой момент.
- UI: в профиле «Уезжаю/не сезон — заморозить подписку» + дата возврата.
- Тесты: продление expires_at ровно на срок заморозки; лимит дней; повторная
  заморозка в том же году.

### 6.5. Симуляция и прогноз дохода (Pro)
Два инструмента на СУЩЕСТВУЮЩЕМ движке (вторая формула не заводится — принцип Ф11.5):
- **Симуляция «что если»:** экран со слайдерами поверх `ProfitabilityCalculator`/
  `OfflineVerdict` (Kotlin, офлайн): цена топлива ±, личная норма ₽/час ±, район
  работы (другая базовая точка) → как меняется вердикт дня/недели. Чистый UI-слой
  над готовым ядром + пресеты («бензин подорожал на 10%»).
- **Прогноз дохода:** по личной истории (`day_metrics` + личные нормы, которые
  уже считаются) — «при твоём темпе этот месяц закроется ~N ₽ чистыми» с
  доверительной вилкой (никакой ML-магии: медиана и квартили по дням недели).
  Сервер: `forecast_service.py`, отдаётся в `/api/home` при Pro.
- Тесты: симуляция не расходится с калькулятором на нулевых сдвигах (парити!);
  прогноз на пустой истории честно молчит («мало данных»), не выдумывает.

### 6.6. Автопарк для газелистов (Pro, ARPU ×5)
**Самая архитектурно-чувствительная фича: RLS изолирует данные по user_id, а
владельцу парка нужна сводка ПО ЧУЖИМ user_id.** Ломать RLS нельзя. Решение:
- Связь: `fleet_members` (§3.1). Приглашение кодом (владелец генерит → водитель
  вводит в приложении → `consent_at` — явное согласие, 152-ФЗ).
- Доступ владельца — НЕ прямыми запросами к чужим строкам, а через **отдельный
  агрегационный сервис** `fleet_service.py`, который работает системным
  соединением (без RLS-GUC), читает ТОЛЬКО дневные агрегаты (чистыми, км, часы,
  визиты) членов парка и отдаёт владельцу сводку. Сырой GPS, адреса и клиенты
  водителя владельцу НЕ видны — только деньги/км/часы дня. Это и приватность,
  и юридическая чистота (мониторинг труда ≠ слежка).
- Водитель видит в профиле «Твою дневную сводку видит {владелец}» + «Выйти из
  парка» одним тапом (уважение к работнику — ДНК продукта).
- Цена: Pro владельца + `fleet_seat_price` ₽/мес за каждого водителя сверх первого
  (серверная настройка; старт: 100 ₽/место). Водителям парка функционал их
  тарифа не меняется (они могут быть free).
- Клиент: экран «Мой парк» у владельца (список машин, сводка дня/недели, сумма).
- Тесты: RLS не протекает (водитель А не видит водителя Б); выход из парка
  мгновенно режет доступ; расчёт цены мест; consent обязателен.

### 6.7. Бенчмарк «ниже медианы города» (Pro, строится ПОСЛЕДНИМ)
Cold-start-фича — включается только с данными (рецензия заседания 2):
- Ночной джоб (`benchmark_service.py`, cron как parking-refresh): агрегирует
  `day_metrics` по ячейкам (город × род занятий × неделя) → медиана и p25/p75
  чистых ₽/час и ₽/км. **k-анонимность: ячейка публикуется только при ≥25
  юзерах** (серверная настройка) — иначе «медиана города» деанонимизирует.
  Агрегаты полностью обезличены (152-ФЗ: обезличенные данные выходят из-под ПДн,
  формулировки согласия свериться при реализации).
- UI (Pro): «Твои 412 ₽/час — это p35 по твоему городу среди мастеров» + позитив
  «до медианы не хватает ~N ₽/час — вот твои 3 самых убыточных типа заказов»
  (склейка с трендами 6.3 — вот где «заработать больше»).
- До накопления данных Pro-карточка честно говорит: «бенчмарк откроется, когда
  в твоём городе будет ≥25 участников» — и это же вирусный крючок («позови
  коллег»). Не выдумывать цифры при малой выборке — принцип «про цену не врать».
- Тесты: ячейка <k молчит; медианы корректны; юзер без города не ломает джоб.

### 6.8. Кандидаты на добор ценности (НЕ делать сейчас, зафиксировать)
- Экспорт истории для налоговой/кредита (совет предлагал в Standard — Джавад не
  включил; вернуться после первых продаж).
- Мини-CRM клиентов (кандидат на крючок удержания из заседания 1) — отдельное
  решение Джавада, в сетке нет.
- B2B-вайт-лейбл, продажа обезличенных данных — «после доказанных оплат»
  (рецензия: расфокус).

## 7. Риски и открытые вопросы

1. **Отъём бесплатного (авто-оптимизация, мультиустройство).** Смягчение:
   grandfathering — всем зарегистрированным до даты T2 подарить Standard на 90
   дней письмом «спасибо, что были с нами до тарифов» (`source='manual'`).
   Одновременно с этим Free ПРИРАСТАЕТ (лента без лимита вместо задуманных ≤5,
   парковка+износ бесплатно) — в коммуникации вести именно с этого.
2. **Free без авто-порядка требует ручного drag в ленте** — если его нет, этап T2
   блокируется на UI-работу; проверить в первом же шаге T2.
3. **Провайдер оплаты для самозанятого** — сверить ЮKassa/альтернативы (§5.1) ДО
   написания кода вебхука; выбор — в Журнал решений.
4. **Окна прибытия (Ф4) при ручном порядке** — пересечение фаз: ETA-цепочка
   должна строиться по ручному порядку тоже; в Ф4 учесть оба режима.
5. **Device-gated клиентская работа** (paywall UI, drag, Room-кеш entitlements) —
   тот же бар, что в MASTER_PLAN: CI-сборка + физическое устройство для QA
   горячих путей (устройство у Джавада есть — задействовать по чек-листу).
6. **Не менять цены в APK.** Всё ценообразование — с сервера; жёстко зашитая
   цена в клиенте = релиз ради каждого изменения и рассинхрон со сторами.
7. **Пересечение с фазами 9–12 MASTER_PLAN:** фазы цикла 2 продолжают ехать по
   плану; тарифные этапы T1–T3 можно вести параллельно (бэкенд-ядро не трогает
   вердикт). В MASTER_PLAN добавить строку в Журнал решений со ссылкой сюда.

## 8. Критерии приёмки всего плана

- Ни в одном тарифе не скрыт ни один компонент себестоимости; free-вердикт
  байт-в-байт равен платному вердикту по одному и тому же заказу.
- Новый юзер: 14 дней полного Standard → мягкая деградация → paywall только на
  рычаге (оптимизация/тренды/синк), правда работает.
- Оплата по ссылке (год/месяц) активирует подписку без ручных действий; вебхук
  идемпотентен (повторная доставка не создаёт вторую подписку).
- Заморозка продлевает подписку ровно на срок паузы.
- Автопарк: владелец видит только дневные агрегаты членов с их согласия; RLS-тест
  «водитель не видит водителя» зелёный.
- Бенчмарк молчит до k=25 и не выдумывает цифры.
- События paywall→оплата видны в аналитике (конверсия измерима).
