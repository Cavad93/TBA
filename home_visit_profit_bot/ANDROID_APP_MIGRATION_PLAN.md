# План переноса функций Telegram-бота в Android-приложение

Цель: сделать Android-приложение **единственным** интерфейсом для всей работы. Конечное состояние — Telegram-бот полностью удалён как устаревший артефакт, а весь функционал (рабочий день, адреса, офис, телемедицина, расходы, отчеты, усталость, GPS, корреляции, настройки и синхронизация) доступен только через Android-приложение и backend REST API.

На переходный период Telegram может оставаться временно рабочим, но каждый его сценарий обязан получить эквивалент в приложении/REST до того, как код Telegram будет вырезан. Финальный этап плана — удалить пакет `app/telegram_bot/`, зависимость `python-telegram-bot` и все привязки к Telegram в точке входа и уведомлениях.

## Принципы

- Итог миграции — приложение без Telegram. Telegram не является резервным каналом в конечном состоянии; он удаляется целиком, поэтому ни одна функция не должна оставаться доступной только через бота.
- Не перегружать главный экран. На первом экране показывать только состояние дня, текущий маршрут и 3-5 основных действий.
- Работать offline-first: критичные данные читаются и вводятся локально даже без сети, а синхронизация с сервером происходит позже.
- Не хранить ФИО пациентов, диагнозы, телефоны и медицинские комментарии.
- Все крупные действия должны быть доступны через понятные кнопки, а не через команды.
- Использовать Material 3: нижняя навигация на телефоне, navigation rail на широких экранах, крупные touch target не меньше 48dp.
- Разделить интерфейс на вкладки, чтобы не превращать главный экран в панель самолета.

## Изученные рекомендации

- Android offline-first: локальный источник данных обязателен для критичных операций, репозитории объединяют локальные и сетевые источники.
  Источник: https://developer.android.com/topic/architecture/data-layer/offline-first
- Android data layer: бизнес-логика и данные должны быть в data/repository слое, отдельно от UI.
  Источник: https://developer.android.com/topic/architecture/data-layer
- Material 3 navigation bar: нижняя навигация подходит для переключения основных разделов на телефоне.
  Источник: https://m3.material.io/components/navigation-bar/overview
- Android adaptive navigation: на больших экранах лучше переходить от bottom navigation к navigation rail.
  Источник: https://developer.android.com/design/ui/mobile/guides/layout-and-content/layout-and-nav-patterns
- Android accessibility: интерактивные элементы должны иметь touch target минимум 48x48dp.
  Источник: https://developer.android.com/guide/topics/ui/accessibility/apps

## Целевая структура приложения

- `Начать день` или `Завершить день`;
- `+ Адрес`;

## Архитектура

### Рекомендуемый путь

Переходить от текущего Java-приложения к Kotlin + Jetpack Compose + Material 3.

Почему:

- текущий Android-клиент минимальный и программно рисует View руками;
- полный функционал станет большим, его сложнее поддерживать в одном `Activity`;
- Compose проще для вкладок, форм, карточек, adaptive layout и состояния экранов.

### Слои

- `ui`: экраны, компоненты, навигация.
- `domain`: бизнес-модели и use cases.
- `data`: Room, API, репозитории, sync queue.
- `sensor`: GPS, акселерометр, гироскоп, дневные агрегаты.
- `network`: REST-клиент к backend API.

### Локальная БД приложения

Использовать Room.

Основные таблицы:

- `work_days`;
- `visits`;
- `office_entries`;
- `telemed_entries`;
- `expenses`;
- `daily_stats`;
- `location_samples`;
- `visit_location_events`;
- `driving_behavior_daily`;
- `burnout_surveys`;
- `fatigue_feedback`;
- `settings`;
- `sync_queue`.

### Синхронизация

Приложение должно уметь работать без Telegram и без постоянной сети.

Модель:

- пользователь вводит данные в приложение;
- запись сохраняется локально;
- событие попадает в `sync_queue`;
- фоновой задачей отправляется на backend;
- backend использует существующую Python-логику расчетов;
- при конфликте выигрывает последняя пользовательская правка, но конфликт логируется.

Минимальные endpoint на старте:

### Этап 10. Модуль настроек (backend + Android) — критический блок — сделано

Статус: реализовано. Backend отдаёт и принимает настройки через `GET/POST /api/settings` (общий `SettingsService` с каталогом полей и валидацией), клиники и телемед-клиники стали настройкой и читаются сервисами кандидата/синхронизации из `settings`, добавлено событие очереди `settings_saved`. Android получил на вкладке «Настройки» экран групп (Экономика, Авто, Дом/старт/финиш, Клиники, Базовые районы, Маршрутизация/OSRM, GPS, Усталость) с чтением `GET /api/settings`, локальным кэшем в Room для offline-просмотра и сохранением через очередь `settings_saved` + немедленную синхронизацию.
